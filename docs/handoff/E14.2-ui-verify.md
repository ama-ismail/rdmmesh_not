## 0. TL;DR за 30 секунд

- **Что появилось в UI:**
  - На `AuditPage` (`/admin/audit`) в `Card.extra` (рядом с заголовком "Audit log")
    — кнопка «Проверить целостность» с `<SafetyCertificateOutlined />` icon.
  - Click → открывает `<Modal>` 640px. При `open=true` идёт
    `useQuery(qk.audit.verify(null, null), () => api.verifyAuditChain())`.
  - Если `verified=true` → AntD `<Result status="success">` с числом
    проверенных записей и range'ем (`#1 … #4`).
  - Если `verified=false` → `<Result status="error">` с указанием
    `first_broken_at`, а ниже `<Descriptions>` с range, reason,
    `expected_hash` (что должно было быть), `stored_hash` (что лежит в БД).
    Hash'и copyable, monospace, мелким шрифтом.
  - Footer modal'а: «Запустить ещё раз» (refetch) + «Закрыть».
- **API layer:**
  - `AuditChainVerifyResult` (snake_case — повторяет backend
    `VerifyChainResponse`).
  - `api.verifyAuditChain(fromId?, toId?)`: GET `/audit/verify-chain` с
    опциональными query-params; пустой call'ом → full chain.
  - `qk.audit.verify(fromId, toId)`: queryKey включает range, чтобы
    full-chain и sub-range кэшировались независимо.
- **i18n** ru/en расширены секцией `audit.verify.*` (17 ключей в каждой
  локали).
- **Bundle:** app-код 29.3 → **30.53 KB gzip** (+1.2 KB на VerifyChainModal +
  i18n + новые AntD-компоненты `<Modal>` / `<Result>` / `<Descriptions>` /
  `<Spin>`). vendor-antd без изменений — все эти компоненты уже были
  подтянуты (Modal — через DiffModal в VersionPage, Result — через
  NotFoundPage, Descriptions — через ConsumerViewDrawer, Spin — через
  ItemsTable loading).
- **Backend не тронут.** Round 1 уже отдаёт `GET /audit/verify-chain`
  (`@RolesAllowed("RDM_ADMIN")`); UI просто его вызывает.

#### Modal — loading

При первом `open` либо `Run again` — `<Spin />` + текст «Проверяем цепочку…».
gcTime=0, staleTime=0 у useQuery: каждое открытие = свежий запрос. Это
сознательно — admin запускает verify как ad-hoc action, не как feed.

### 1.3. VerifyChainModal — детально

```tsx
function VerifyChainModal({ open, onClose }) {
  const verify = useQuery({
    queryKey: qk.audit.verify(null, null),
    queryFn: () => api.verifyAuditChain(),
    enabled: open,
    staleTime: 0,
    gcTime: 0,
  });

  return (
    <Modal open={open} onCancel={onClose} width={640} footer={...}>
      {verify.isFetching && <Spin + "Проверяем цепочку…" />}
      {!verify.isFetching && verify.error && <Alert type="error" />}
      {!verify.isFetching && verify.data && <VerifyChainResultView ...>}
    </Modal>
  );
}
```

`enabled: open` важно — query не стартует, пока modal закрыт; иначе на
каждом mount AuditPage уходил бы лишний REST-вызов. После закрытия Modal'а
gcTime=0 удаляет cached entry сразу — повторный open даст fresh fetch.

**Errors.** `verify.error` — `ApiError` от `apiFetch`:
- 403 → admin потерял роль между логином и кликом. UI показывает
  `403: <message>`.
- 400 → невалидный range (не случается, потому что UI не передаёт
  параметры). Показывается как `400: …`.
- 500 → backend упал. Показывается как `500: …`.

### 1.4. VerifyChainResultView — детально

`verified=true` → AntD `<Result status="success">` с интерполяцией
`{{checked}}`, `{{from}}`, `{{to}}` в subTitle.

`verified=false` → `<Result status="error">` + `<Descriptions>` с
четырьмя rows: range, reason, expected_hash, stored_hash. Hash'и —
copyable (`<Typography.Text copyable>`) для удобного pasting в Jira/Slack
при инциденте.

`reason` приходит из backend'а с текстом типа
`"entry_hash mismatch: recomputed=abc..., stored=def..."` либо
`"prev_hash mismatch: expected=abc..., stored=def..."`. UI отображает
as-is, не пытается парсить.

## 2. Контракт: что UI ожидает от backend'а

### 2.1. Без изменений с round 1

| Endpoint | Метод | Где используется | Auth |
|---|---|---|---|
| `/api/v1/audit/verify-chain` | GET | VerifyChainModal | `RDM_ADMIN` |

Response — `AuditChainVerifyResult` (snake_case, см. backend §1.6
handoff E14-compliance.md):

```ts
interface AuditChainVerifyResult {
  from: number;
  to: number;
  checked: number;
  verified: boolean;
  first_broken_at?: number | null;
  reason?: string | null;
  expected_hash?: string | null;
  stored_hash?: string | null;
}
```

При `verified=true` поля `first_broken_at/reason/expected_hash/stored_hash`
— `null` (backend `@JsonInclude(NON_NULL)`). UI читает их через optional
chaining (`result.expected_hash &&`), так что отсутствие поля не вызывает
TypeError.

### 2.2. Без изменений (round 1 → round 2)

PATCH / GET остальных audit-endpoint'ов — без правок. UI на новых
полях `payload_canonical/prev_hash/entry_hash` в `audit_log` не зависит
— они exposed только через verify-эндпоинт, AuditEntryDto их по-прежнему
не возвращает (E11.2d виду не нужно).

## 3. Что осталось доделать — мягкие follow-up'ы

Ничего не блокирует следующий round. Список того, к чему стоит вернуться:

1. **Sub-range verify через DatePicker.** Сейчас UI всегда вызывает
   полный chain. Для compliance-аудитора, который смотрит «инцидент с
   30.03.2026 по 02.04.2026», полезен range. Решение — в `Modal` добавить
   ID-range или DatePicker (с конвертацией в id'ы через
   `findChainRange`-эквивалент на backend — потребует новый endpoint
   либо параметр `from_ts`/`to_ts`). На пилоте full-chain быстро.

2. **Авто-verify cron-job + email-alerting.** Compliance team скорее
   всего захочет «verify раз в 6 часов, при разрыве — оповестить admin'ов
   через Slack/email». Это уже не UI — отдельный background-worker
   либо OS-level cron. handoff E14-compliance.md §3 #4 (chain breakage
   policy).

3. **Verify-action audit-event.** REST-вызов verify сейчас сам не пишется
   в audit — мы читаем audit, не пишем. Если compliance хочет видеть
   «кто и когда запускал verify» — нужен AuditVerifyDomainEvent (handoff
   E14-compliance.md §3 #8). Round 11 territory.

4. **Длинные chain'ы и UX.** При chain'е >100K rows verify может занять
   несколько секунд. Сейчас в Modal только `<Spin>` без прогресса —
   admin не видит, насколько процесс продвинулся. Решение: переключить
   на streaming progress (server-sent events) либо batched verify (по
   10K rows + накопительный счётчик в UI). На пилоте этого не нужно.

5. **Tree-shake unused i18n keys.** Если в проекте используется
   i18n-tree-shake (мы не используем — `index.ts` импортирует обе локали
   целиком), 17 новых ключей добавили ~1 KB к bundle'у. Не оптимизировать.

6. **AntD 5 migration.** Сейчас в проекте смесь AntD 4.x паттернов
   (handoff E1) и AntD 5 (`dayjs` импортирован напрямую, новые
   компоненты `<Descriptions.Item>` имеют AntD 5 API). Это работает,
   но при будущем upgrade AntD 6 — пересмотреть. Не блокирует ничего.

7. **VerifyChainResultView вынести в отдельный файл.** Сейчас он
   определён внизу `AuditPage.tsx` рядом с `PayloadView`/`trim`. Если
   компонент начнёт расти (например, custom rendering на конкретный
   reason-type) — вынести в `src/components/VerifyChainResultView.tsx`.

## 4. Технический долг и решения

| Что | Где | Когда снять / следующий шаг |
|---|---|---|
| `qk.audit.verify(null, null)` фиксирован как full-chain | `VerifyChainModal` | По §3 #1 — добавить range-параметры в querykey, когда появится UI range-selector. |
| `staleTime=0 / gcTime=0` у verify | `VerifyChainModal` | by-design — admin-action, не feed. Не кэшировать. |
| VerifyChainResultView и VerifyChainModal в `AuditPage.tsx` | UI | by-design на M-объёме; вынести при росте (см. §3 #7). |
| Backend reason — plain string, UI его не парсит | `VerifyChainResultView` | Если custom UX потребуется — backend должен отдавать `reason_code: "ENTRY_HASH_MISMATCH" \| "PREV_HASH_MISMATCH"` отдельно. Сейчас single message достаточен. |
| Verify-кнопка всегда в Card.extra, даже при `q.isLoading` основной таблицы | `AuditPage` | by-design — verify независим от listing-загрузки. |
| `404` на verify-chain (пустой журнал) | `AuditResource.verifyChain` | Backend возвращает `200 OK` с `from=0, to=0, checked=0, verified=true` — UI отобразит «Цепочка цела, проверено: 0 (#0 … #0)». Это семантически OK. |
| `403`-fallback показывается внутри modal'а, а не на странице | `VerifyChainModal` | by-design — frontend-guard на странице уже ловит обычный случай; modal-фоллбек для race-condition при потере роли. |

## 6. Открытые вопросы (актуальны для команды банка)

Без изменений с E14-compliance.md, плюс UI-specific:

1–36 — без изменений (см. `E14-compliance.md` §7).
37. **Phrasing.** "Цепочка цела" vs "Целостность подтверждена" vs
    "Журнал верифицирован". Сейчас выбрано "Цепочка цела" — короче,
    technically accurate. Compliance team может предпочесть более
    формальный язык. Решить при первом review с командой compliance.
38. **Кнопка «Run again» vs «Refresh».** Сейчас "Запустить ещё раз" /
    "Run again". Альтернатива — auto-refetch при открытии каждой
    отдельной сессии modal'а (что и так происходит благодаря
    `gcTime=0`). Дополнительная кнопка — это явный signal для
    повторного запуска без закрытия modal'а; полезно при подозрении на
    race-condition. На пилоте — оставить кнопку.
39. **Modal width 640px.** Помещает все expected_hash/stored_hash (64
    chars) в одну строку при типичных screen-resolution'ах. На
    маленьких экранах (планшет 768px) hash переносится. Acceptable;
    альтернатива — `width="min(640px, 90%)"` — не делать без жалобы.
