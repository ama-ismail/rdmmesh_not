## 1. Что сделано

### 1.1. Keycloak realm — новый user + группа + роль

`docker/keycloak/realms/realm-bank.json`:

```json
"groups": [
  { "name": "RDM_ADMIN" }, { "name": "RDM_AUTHOR" }, ...,
  { "name": "RDM_AUDITOR" }, { "name": "RDM_CONSUMER" }
],
"roles": {
  "realm": [
    ...,
    { "name": "RDM_AUDITOR",
      "description": "Read-only доступ к audit-журналу и verify-chain (compliance team).
                      Не имеет прав admin'а — не управляет subscriptions, не запускает
                      closure rebuild." },
    { "name": "RDM_CONSUMER", ... }
  ]
},
...
"users": [
  ...,
  {
    "username": "dev-auditor",
    "enabled": true,
    "email": "dev-auditor@bank.local",
    "firstName": "Dev", "lastName": "Auditor",
    "credentials": [{ "type": "password", "value": "dev", "temporary": false }],
    "groups": ["RDM_AUDITOR"]
  }
]
```

Пароль `dev` — тот же шаблон, что у других dev-юзеров (admin/author/steward/owner).
Realm применяется через `KEYCLOAK_IMPORT` при первом старте dev-стека; **после
изменения JSON нужен `docker compose down -v` либо ручной import в Keycloak Admin
UI**, иначе старый realm в волюме `keycloak-data` будет использоваться без новой
группы.

### 1.2. Backend identity — без правок

`JwtValidator.Resolved.groups` → `AuthenticatedUser(baseRoles=groups)` →
`RdmmeshPrincipal.hasRole(role) → baseRoles.contains(role)` →
`RoleAuthorizer.authorize(principal, role, ctx)`. Никакого whitelist'а в коде
нет — любая группа из JWT передаётся as-is. Это сознательное решение E2
handoff'а: realm — единственный источник истины для ролей, backend не
дублирует список.

Следствие: чтобы вырубить роль на backend'е (если когда-то понадобится emergency
disable без передеплоя Keycloak'а) — нужен будет либо deny-list в
`RoleAuthorizer`, либо feature-flag в config. Это пока не нужно.

### 1.3. AuditResource — class-level @RolesAllowed расширен

`rdmmesh-audit/src/main/java/bank/rdmmesh/audit/resource/AuditResource.java`:

```diff
-@RolesAllowed("RDM_ADMIN")
+@RolesAllowed({"RDM_ADMIN", "RDM_AUDITOR"})
 public final class AuditResource {
```

Class-level annotation наследуется обоими методами:
- `GET /api/v1/audit` (listing с фильтрами, E11.2d) — доступен обоим.
- `GET /api/v1/audit/verify-chain` (E14 round 1) — доступен обоим.

Javadoc class'а и метода `verifyChain` обновлён, чтобы будущий читатель знал,
что compliance может verify запускать, а не только admin.

### 1.4. SubscriptionsResource — намеренно НЕ трогали

`rdmmesh-publishing/.../SubscriptionsResource.java`: остаётся
`@RolesAllowed("RDM_ADMIN")`. Auditor не должен управлять webhook'ами
consumer'ов — это конфигурация, не journal.

То же относится к:
- `ClosureAdminResource` (E13.3, `POST /versions/{id}/closure/rebuild`) — admin-only.
- Bootstrap-domain endpoint (handoff E7) — admin-only.

Если compliance team нужно увидеть `subscriptions` (например, для проверки кто
куда шлёт payload'ы) — отдельный read-only endpoint
`GET /subscriptions/read-only` под `RDM_AUDITOR`. Это V1+ либо отдельный round.

## 3. Что осталось доделать — мягкие follow-up'ы

Ничего не блокирует следующий round. Список того, к чему стоит вернуться:

1. **AD-mapping в prod.** Сейчас realm-конфиг описывает только dev-стек.
   В prod compliance-роль приходит из AD-group `CN=rdm-auditor,OU=Groups,DC=bank,DC=local`
   (имя согласовать с IAM-team). Keycloak LDAP federation должен mapping'ить эту
   AD-группу в realm-group `RDM_AUDITOR`. Реалм-конфиг — отдельный артефакт
   (`docker/keycloak/realms/realm-bank-prod.json` либо Terraform module). На
   пилоте — обсудить с командой эксплуатации перед prod-deploy.

2. **Доступ к `GET /subscriptions` для auditor'а.** Compliance team может
   попросить «покажите, какие webhook'и сейчас зарегистрированы» (для проверки
   утечки данных). Решение — read-only метод `GET /subscriptions` под
   `@RolesAllowed({"RDM_ADMIN","RDM_AUDITOR"})`, а POST/DELETE остаются
   admin-only. Сделать когда compliance явно попросит.

3. **Domain-scoped audit.** Сейчас auditor видит весь журнал. Для крупного
   банка compliance-команда может быть разделена по доменам (например,
   compliance-risk видит только Risk-журналы). Решение — фильтр на backend
   `WHERE aggregate_id ∈ subtree_of(user.domains)` + claim в JWT
   `audit_domains: ["risk_*"]`. V1+ feature.

4. **Audit trail of audit access.** «Кто и когда читал audit?» Сейчас REST-вызов
   не пишется в сам журнал — мы читаем audit, не пишем. Compliance может
   захотеть meta-audit. Решение — handoff E14-compliance.md §3 #8: эмитить
   `AuditAccessDomainEvent` из AuditResource'а через тот же EventBus.

5. **HEAD-of-chain hash в UI.** Удобно показать compliance-юзеру «последний
   entry_hash» как «snapshot identity» — это даёт possibility сравнивать
   между разными восстановлениями БД. UI-feature, ~1 KB bundle.

6. **Rate-limit на verify-chain.** Полный chain verify может быть нагрузочным
   на крупном journal'е. Auditor может теоретически DDoS'ить backend
   повторными вызовами. Не критично на пилоте; добавить `RateLimiterFilter`
   на верхнем уровне когда дойдёт до prod-deploy.

7. **Тест integration для @RolesAllowed.** Сейчас покрытие — backend ArchUnit
   + ручной smoke. Полноценный testcontainer-test (JwtAuthenticator + RoleAuthorizer
   с реальным RealmDescriptor) — мягкий debt, как у E2/E5/E10. Закрыть в
   round 9 (testcontainers IT infrastructure).

## 4. Технический долг и решения

| Что | Где | Когда снять / следующий шаг |
|---|---|---|
| Backend identity-pipeline без role whitelist'а | `JwtValidator` / `KeycloakIdentityPort` | by-design — realm authoritative. Не добавлять без явной причины. |
| Group и role одновременно в realm с тем же именем | `realm-bank.json` | Keycloak паттерн: group для membership, role для permission. RDM использует только group → JWT mapping. |
| AppLayout adminChildren conditional через push | `AppLayout.tsx` | Чище — переписать через `.filter()`. Micro-cleanup. |
| `canViewAudit` дублируется между AuditPage и AppLayout | UI | Хук `useCanViewAudit()` в `auth/`. Стоит сделать если число access-уровней вырастет. V1+. |
| `dev-auditor` пароль `dev` — тот же что у других dev-юзеров | realm | Acceptable для dev-стека. В prod все пароли через LDAP. |
| `RDM_AUDITOR` description в realm-роли — на русском | `realm-bank.json` | Остальные роли тоже на русском; consistent. |
| VerifyChainModal видим RDM_AUDITOR'у | `AuditPage` | by-design — это и есть основная feature для compliance. |

## 6. Открытые вопросы (актуальны для команды банка)

Без изменений с E14-compliance.md, плюс round 3 specific:

1–39 — без изменений (см. `E14.2-ui-verify.md` §6).
40. **Naming convention в prod AD.** Realm-конфиг dev'а описывает группу
    как `RDM_AUDITOR` (uppercase, underscore). В prod AD это могут быть
    `CN=rdm-auditor,OU=Groups,DC=bank,DC=local` либо `rdm_auditor` либо
    `RDMA_AUDIT`. Решение зависит от IAM-team стандарта. Mapping в Keycloak
    federation должен превратить AD-group в realm-group `RDM_AUDITOR`
    (имя в JWT claim фиксировано — на это завязан backend `@RolesAllowed`).
41. **RDM_AUDITOR ↔ asset-level ownership.** SPEC §2.1 говорит, что
    asset-level роли (Owner/Steward/Expert per CodeSet) приходят из OM.
    RDM_AUDITOR — глобальная функциональная роль через AD/Keycloak,
    не per-asset. Один и тот же человек может быть и `RDM_AUDITOR`
    глобально, и `OWNER` на конкретный CodeSet. Это legal — две независимые
    оси. Compliance должны подтвердить, что такое совмещение допустимо
    (vs «auditor не может быть owner»-policy).
42. **«expert ⊇ auditor»-policy?** Если бизнес считает, что любой OM-expert
    автоматически имеет audit-доступ (типично — эксперты часто и есть
    compliance) — нужен mapping `OM expert → RDM_AUDITOR` через OM webhook.
    Сейчас этого нет; expert на CodeSet даёт только asset-role в RDM.
    Решить с командой compliance.
