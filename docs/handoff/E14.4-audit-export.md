## 0. TL;DR за 30 секунд

- **Backend (rdmmesh-audit):**
  - **`AuditLogDao.findExportPage(...)`** — новый метод. Те же фильтры что у
    `findPaged`, но с двумя отличиями: `WHERE id <= :snapshotMaxId` (snapshot-
    consistency: concurrent INSERT'ы в audit_log поверх snapshot'а не попадают
    в выгрузку) и `ORDER BY id ASC` (стабильный порядок по BIGSERIAL PK).
    Возвращает новый record `ExportRow` с hash-chain полями (prev_hash,
    entry_hash, payload_canonical) — compliance-аудитор может оффлайн
    пересчитать chain без обращения к verify-endpoint'у.
  - **`AuditExportWriter`** (новый, pure) — два формата:
    - **CSV** по RFC 4180: header-row + N data-rows, quote-only-if-needed,
      внутренние `"` удваиваются. 11 колонок: id, event_id, event_type,
      aggregate_type, aggregate_id, actor, occurred_at, payload,
      payload_canonical, prev_hash, entry_hash.
    - **NDJSON**: одна валидная JSON-row на строку (с trailing `\n`).
      `payload` и `payload_canonical` идут **inline как embedded JSON-объекты**,
      не escape-string'и — это упрощает `jq .payload.foo` на стороне аудитора.
    - Класс stateless, 13 unit-тестов покрывают edge cases (escaping, null,
      специальные символы).
  - **`AuditResource.export(...)`** — новый `GET /api/v1/audit/export`:
    - `@RolesAllowed({"RDM_ADMIN","RDM_AUDITOR"})` через class-level annotation
      (наследуется с round 3).
    - Query: те же фильтры что и `/audit` (event_type, aggregate_type,
      aggregate_id, actor, from, to, q) + `format=csv|ndjson` (default csv).
    - Response: `StreamingOutput` с pagination-loop'ом по 1000 rows.
      `Content-Disposition: attachment; filename="audit-20260512-153045.csv"`
      (UTC timestamp). MIME `text/csv` либо `application/x-ndjson`.
    - Header `X-RDM-Snapshot-Max-Id: <id>` — снимок-id, по которому был сделан
      export; полезен для audit-trail downloads (V1+ либо round 11).
- **Frontend (rdmmesh-ui):**
  - **`apiFetchRaw(path, init?)`** — новый helper в `api/client.ts`. То же что
    `apiFetch` (JWT прокидывает), но возвращает `Response` без auto-парсинга —
    caller сам решает что делать (blob, stream, text).
  - **`api.downloadAuditExport(filters, format)`** — fetch → blob → URL.createObjectURL
    → synthetic `<a download>` click → revokeObjectURL. Filename берётся из
    `Content-Disposition` header'а либо fallback'ом строится локально.
  - **`AuditPage` `Card.extra`:** теперь `<Space>` с двумя кнопками. Левая —
    `<Dropdown.Button>` «Скачать» с menu CSV / NDJSON; click → spinner на
    кнопке + downloadAuditExport; `message.success` либо `message.error`.
    Правая — verify-button (без изменений с round 2).
- **i18n** ru/en расширены секцией `audit.export.*` (6 ключей).
- **Bundle:** app-код 30.60 → **31.42 KB gzip** (+0.82 KB на download utility +
  Dropdown + i18n). vendor-antd 368.92 → 369.91 KB gzip (+1 KB на

## 1. Что сделано

### 1.1. `AuditLogDao.findExportPage` — снапшот-консистентный pagination

```sql
SELECT id, event_id, event_type, aggregate_type, aggregate_id, actor,
       occurred_at, payload::text AS payload, payload_canonical,
       prev_hash, entry_hash
  FROM audit.audit_log
 WHERE id <= :snapshotMaxId        -- ← снапшот!
   AND (:eventType     IS NULL OR ...)
   AND (:aggregateType IS NULL OR ...)
   ...
 ORDER BY id ASC                   -- ← BIGSERIAL PK index scan
 LIMIT :limit OFFSET :offset
```

**Почему `WHERE id <= snapshotMaxId`.** Без него concurrent INSERT'ы в audit
поверх pagination-loop'а получают сдвиг offset'ов: одна row может оказаться на
двух соседних страницах либо пропасть. На пилоте трафик в audit ≤ десятков
записей в час — теоретически безопасно, но compliance не должен зависеть от
luck'а. Snapshot'ный id фиксируется в `AuditResource.export` через
`findMaxId().orElse(0L)` **до** начала streaming'а.

**Почему `ORDER BY id ASC` а не `occurred_at DESC` (как в `findPaged`).** Для
viewer'а нужна latest-first сортировка, для export'а — стабильный порядок и
быстрый index scan. BIGSERIAL `id` монотонно растёт; index по PK уже есть.

**Новый record `ExportRow`** (11 полей). Включает hash-chain (`prev_hash`,
`entry_hash`, `payload_canonical`) — это полное содержимое строки журнала.
Compliance может оффлайн пересчитать SHA-256 chain без обращения к
verify-endpoint'у.

### 1.2. `AuditExportWriter` — два формата

**`writeCsvHeader(out)`**:
```
id,event_id,event_type,aggregate_type,aggregate_id,actor,occurred_at,payload,payload_canonical,prev_hash,entry_hash\r\n
```

**`writeCsvRow(out, row)`**: каждая cell через `writeCsvCell`. Логика
quoting'а RFC 4180:
- Пустая cell → ничего не пишется (`,,,`).
- ASCII-строка без `,`, `"`, `\n`, `\r` → as-is.
- Иначе → обёрнута в `"..."`, внутренние `"` удвоены (`"` → `""`).

**`writeNdjsonRow(out, row)`**: ручной JSON-builder через StringBuilder. Не
использует Jackson, потому что:
1. Нужен фиксированный порядок ключей (stable diff между экспортами).
2. `payload` уже валидный JSON-string из jsonb (через `::text`), нужно его
   embed-нуть **без re-сериализации** (`"payload":<raw_json>`), что Jackson
   сделать без CustomSerializer'а не умеет.
3. Зависимость на ObjectMapper только под 11 полей — overkill.

Embedded payload пример:
```json
{"id":4,"event_id":"...","event_type":"VERSION_PUBLISHED",...,"payload":{"version":"0.1.0","content_hash":"..."},"payload_canonical":{...},"prev_hash":"...","entry_hash":"..."}
```

Это полезно для `jq`: `jq '.payload.version'` работает напрямую. Если бы
payload был escape-строкой, аудитор должен был бы дополнительно
`fromjson`-ить.

### 1.3. `AuditResource.export` — endpoint

```java
@GET
@Path("/export")
public Response export(
        @Auth RdmmeshPrincipal principal,
        @QueryParam("event_type") String eventType, ...,
        @QueryParam("format") String formatRaw) {
    // 1. Resolve filters (UUID, timestamp parsing — переиспользует существующие
    //    parseUuid/parseInstant из round 1).
    // 2. Validate format: csv|ndjson.
    // 3. snapshotMaxId = findMaxId().orElse(0L).
    // 4. StreamingOutput с pagination loop по 1000 rows.
    // 5. Headers: Content-Disposition, X-RDM-Snapshot-Max-Id.
}
```

**Pagination loop в `writeExport`:**
```java
try (Writer w = new BufferedWriter(new OutputStreamWriter(output, UTF_8))) {
    if ("csv".equals(format)) AuditExportWriter.writeCsvHeader(w);
    long offset = 0;
    while (true) {
        List<ExportRow> page = jdbi.withExtension(..., dao ->
            dao.findExportPage(snapshotMaxId, ..., EXPORT_PAGE_SIZE, offset));
        if (page.isEmpty()) return;
        for (ExportRow r : page) writeRow(w, r, format);
        if (page.size() < EXPORT_PAGE_SIZE) return;
        offset += EXPORT_PAGE_SIZE;
    }
}
```

**Heap footprint:** одна страница ≤ 1000 row × ~1 KB payload ≈ 1 MB. Не зависит
от размера выгрузки — миллионы rows не разрушат сервер.

**Filename:** `audit-20260512-153045.csv` где timestamp — UTC момент старта
export'а. Это даёт хороший sort-order в файловой системе аудитора.

### 1.4. `apiFetchRaw` — frontend helper

```ts
export async function apiFetchRaw(path: string, init?: RequestInit): Promise<Response> {
  const token = await getToken();
  const headers = new Headers(init?.headers ?? {});
  if (token) headers.set("Authorization", `Bearer ${token}`);
  const res = await fetch(buildUrl(path), { ...init, headers });
  if (!res.ok) throw new ApiError(res.status, ..., body);
  return res;
}
```

Отличия от `apiFetch`:
- Не ставит `Accept: application/json` по умолчанию (download endpoint может
  возвращать любой MIME).
- Не парсит body — caller сам решает (blob, stream, text).
- На non-2xx читает body как JSON или text и бросает `ApiError` — для
  consistency с остальным API layer'ом.

### 1.5. `api.downloadAuditExport(filters, format)` — blob download

```ts
const res = await apiFetchRaw(`/audit/export?${params}`);
const blob = await res.blob();
const filename = parseFilenameFromContentDisposition(
    res.headers.get("content-disposition")
) ?? `audit-${nowStamp()}.${format}`;

const url = URL.createObjectURL(blob);
try {
    const a = document.createElement("a");
    a.href = url; a.download = filename;
    document.body.appendChild(a);
    a.click(); a.remove();
} finally {
    URL.revokeObjectURL(url);
}
```

**Почему blob а не `window.location = url`.** `window.location` не прокидывает
`Authorization` header — нужен либо cookie-auth, либо blob. Blob — стандартный
SPA-паттерн.

**`parseFilenameFromContentDisposition`** — простой regex
`/filename="([^"]+)"/i`. RFC 6266 поддерживает encoded form (`filename*=UTF-8''...`)
— backend пока не использует, поэтому regex достаточен.

**Query parameters:**
- `format` — `csv` (default) либо `ndjson`. Любое другое → 400.
- `event_type`, `aggregate_type`, `aggregate_id`, `actor`, `from`, `to`, `q` —

**Response headers:**
- `Content-Type: text/csv; charset=UTF-8` либо `application/x-ndjson; charset=UTF-8`.
- `Content-Disposition: attachment; filename="audit-YYYYMMDD-HHmmss.<ext>"`.
- `X-RDM-Snapshot-Max-Id: <long>` — id-граница snapshot'а.

**Response body:**
- **CSV** — RFC 4180, 11 колонок, header + N data-rows.
- **NDJSON** — одна валидная JSON-row на строку, разделитель `\n`.

### 2.2. Алгоритм snapshot consistency

```
snapshot_max_id = SELECT max(id) FROM audit.audit_log  -- atomically at request start

for offset in {0, 1000, 2000, ...}:
    page = SELECT ... FROM audit.audit_log
            WHERE id <= snapshot_max_id
              AND <filters>
            ORDER BY id ASC
            LIMIT 1000 OFFSET offset
    if page is empty: return
    write each row
    if |page| < 1000: return
```

Это даёт строгий repeatable-read snapshot для всей выгрузки даже без явного
`SET TRANSACTION ISOLATION LEVEL REPEATABLE READ` (каждая страница в своей
короткой транзакции, но фильтр гарантирует консистентность).

### 2.3. Без изменений

`/audit` listing (round 1) и `/audit/verify-chain` (round 1) — без правок.
Frontend `AuditPage` фильтры пробрасываются и в listing, и в export.

## 3. Что осталось доделать — мягкие follow-up'ы

Ничего не блокирует следующий round. Список того, к чему стоит вернуться:

1. **Audit-trail для downloads.** Сейчас `GET /audit/export` не пишется в сам
   журнал — мы читаем, не пишем. Compliance может захотеть «кто и когда что
   скачал». Решение — `AuditExportDomainEvent` через EventBus → audit подхватит
   (round 11 territory). Header `X-RDM-Snapshot-Max-Id` уже даёт частичный
   audit-trail через access-log сервера.

2. **XLSX формат.** CSV открывается в Excel, но даты как text, не как Date
   cell-type. Аудитор должен ручками cast'нуть. Можно подключить
   `apache.poi:poi-ooxml` (тяжёлая dep, ~10 MB jar) и стримить XLSX. На пилоте
   не оправдано; V1+ если compliance попросит.

3. **Scheduled export в S3.** Compliance может попросить «каждую ночь
   полный snapshot CSV в S3 immutable bucket». Backend cron-job + S3 client.
   Это **round 10** roadmap'а (handoff E14-compliance.md §5).

4. **Excel-friendly CSV: BOM + semicolon delimiter.** Excel ожидает UTF-8 BOM
   + `;` delimiter в EU-локалях. Сейчас выгружаем чистый UTF-8 + `,` (RFC 4180).
   Excel прочтёт, но некрасиво. Можно добавить query-param `excel_friendly=true`.
   Не делать без жалобы.

5. **Streaming через true `Stream<T>` Jdbi.** Сейчас pagination loop по 1000.
   Альтернатива — `@SqlQuery` + `Stream<T>` с server-side cursor (Postgres
   JDBC `setFetchSize(N)` + autocommit=false). Технически чище, но требует
   open-handle pattern в Resource'е. Pagination проще и предсказуемее для
   пилотных объёмов. Migrate'нуть когда первый export станет >100k rows.

6. **Rate-limit на export.** Аудитор теоретически может DDoS'ить backend
   повторными вызовами полного export'а. Не критично на пилоте; добавить
   `RateLimiterFilter` на верхнем уровне когда дойдёт до prod-deploy. Альтернатива
   — отдельная `RDM_AUDITOR`-quota в Keycloak: ≤ 10 export'ов в час.

7. **Streaming response в UI.** Сейчас весь blob держится в JVM heap'е сервера
   (через Jersey buffering) **и** в memory браузера (через `await res.blob()`).
   Для очень больших экспортов (>100 MB CSV) лучше использовать
   `ReadableStream` API: `res.body.pipeTo(...)` напрямую в файл через
   `showSaveFilePicker` (File System Access API). Сейчас 90% браузеров не
   поддерживают это; V1+.

8. **`Stream<ExportRow>` тест через testcontainers.** Сейчас покрытие — unit
   на writer'ы + ручной smoke. Backend integration (pagination loop, snapshot
   correctness под concurrent writes) — тот же мягкий debt, что у других
   эпиков. Round 9.

## 4. Технический долг и решения

| Что | Где | Когда снять / следующий шаг |
|---|---|---|
| Pagination loop вместо честного streaming через `Stream<T>` | `AuditResource.writeExport` | Acceptable для MVP; см. §3 #5. |
| BufferedWriter без explicit flush в pagination loop | `writeExport` | OK — Jersey закрывает `OutputStream` после StreamingOutput; try-with-resources на Writer'е делает flush+close. |
| `Content-Disposition` filename без RFC 6266 encoding | `AuditResource.export` | Имя `audit-YYYYMMDD-HHmmss.csv` чистый ASCII — encoding не нужен. Если когда-то добавится пользовательский filename — переключиться на `filename*=UTF-8''...`. |
| `payload_canonical` дублирует payload в export | `ExportRow` | by-design — compliance должен иметь обе формы. payload — human-readable, payload_canonical — для verify chain. |
| `X-RDM-Snapshot-Max-Id` header пока не используется UI'ем | `AuditResource.export` | Полезно для audit-trail downloads. V14 round 11. |
| `nowStamp()` на frontend'е использует UTC | `endpoints.ts` | Match'ит backend Filename, но локальная зона юзера была бы более intuitive. UTC соблюдается потому что Content-Disposition обычно используется первым. |
| `apiFetchRaw` бросает ApiError на non-2xx, но не парсит тело как JSON если backend ответил 400 c JSON | `client.ts` | Парсит — но только если `Content-Type: application/json`. Backend нашего export'а на 400 отдаёт plain-text — это OK. |
| Dropdown.Button с двумя items вместо отдельных кнопок CSV/NDJSON | UI | UX-trade-off: одна кнопка с выбором меньше шумит. Если compliance reports «всегда CSV» — переключиться на одну кнопку. |
| Export НЕ имеет own audit-event | `AuditExportDomainEvent` | Round 11 (handoff E14.4 §3 #1). |

## 6. Открытые вопросы (актуальны для команды банка)

Без изменений с E14.3, плюс round 4 specific:

1–42 — без изменений (см. `E14.3-rdm-auditor.md` §6).
43. **CSV delimiter.** Сейчас `,` (RFC 4180). Excel в RU-локали ожидает `;`.
    Решение — auto-detect по `Accept-Language` либо явный query-param
    `excel_friendly=true`. На пилоте не критично — Excel сам предложит
    выбрать разделитель. Решить если compliance team скажет «неудобно».
44. **Размер max-export.** Сейчас нет лимита. Compliance может ожидать
    «все 7 лет retention'а» в один download. На retention'е 7 лет это могут
    быть сотни MB CSV. Решение — два варианта: либо без лимита (как сейчас,
    risk timeout'ов в reverse-proxy), либо страничный download через дату
    (например, «по году»). На пилоте без лимита OK.
45. **Excel BOM.** Excel требует UTF-8 BOM (EF BB BF) для корректной
    русской кодировки. Сейчас не добавляем — Excel правильно угадывает
    UTF-8 (modern versions ≥ 2016). Если compliance team на старом Excel
    — добавить BOM перед `id,event_id,...`-header'ом одной строкой кода.
46. **Verify-export workflow.** Compliance скачивает CSV → проверяет
    hash-chain → … но как? Нужен offline-tool типа `rdmmesh-audit-verify
    < audit.csv` либо UI кнопка "Загрузить CSV → verify". Сейчас цепочку
    можно проверить только через online verify-endpoint. Inline verify
    на скачанном файле — round 11 либо отдельный CLI-tool.
