## 1. Что сделано

### 1.1. Зависимости

```diff
+ "@tanstack/react-query": "^5.59.0"     // dependencies
```

В сумме `+3 пакета` (`@tanstack/query-core`, `@tanstack/react-query`,
`@tanstack/query-devtools` подтянулся опционально). Devtools НЕ подключены к
`<App/>` — добавляются отдельным компонентом, не критично для round 2a; включить
в round 2b при первом серьёзном debugging'е cache'ов.

### 1.2. `src/api/queryClient.ts` — новый

Один глобальный `QueryClient`:

```ts
defaults.queries = {
  retry: (count, err) =>
    err instanceof ApiError && err.status < 500 ? false : count < 1,
  staleTime: 30_000,
  refetchOnWindowFocus: false,
};
defaults.mutations = { retry: false };
```

Логика retry: 4xx (включая 401/403/404/409/422) — пользовательские ошибки, нет
смысла повторять. 5xx — один повтор. Mutations не повторяем никогда (caller сам
решает — особенно для transition'ов, где повторный вызов может попасть на уже
случившийся переход и получить 409 от state machine).

`qk.*()` — централизованная фабрика query keys. Keys организованы как иерархии
для дешёвой `invalidateQueries({ queryKey: prefix })`:

```
qk.versions.byCodeset(id)        ← список версий codeset'а
qk.versions.one(id)              ← одна версия
qk.versions.items(id, page, size)← страница items
qk.versions.history(id)
qk.tasks.my()
qk.codesets.one(id)
…
```

### 1.3. `src/api/useApi.ts` — переписан

Старая сигнатура была `useApi(loader, deps[])` с самописным useEffect/useState.
Новая — `useApi(loader, queryKey)`, под капотом `useQuery`. Возврат тот же
`{loading, error, data}` — все вызовы `<Loader {...state}>` остались как были.

Замена в существующих pages: вторым аргументом теперь идёт `qk.*()` вместо
`[deps]`. Если деп изменится (например, `[id]` → `qk.codesets.one(id)`), key тоже
изменится → React Query сделает refetch. Поведенческой разницы нет, но теперь
mutation hooks могут попадать в этот key через `queryClient.invalidateQueries`.

### 1.4. `src/api/endpoints.ts`

Read-side `api.*` без изменений. Добавлен `apiMutations.*` для write'а:

| Метод | URL | E-эпик |
|---|---|---|
| `createDraft(codesetId, body)` | `POST /versions/by-codeset/{id}` | E4 |
| `deleteDraft(versionId)` | `DELETE /versions/{id}` | E4 |
| `createItem(versionId, item)` | `POST /versions/{id}/items` | E4 |
| `transition(versionId, req)` | `POST /versions/{id}/transitions` | E5 |

`createDraft` принимает опциональный `version` (semver) — если не задан, backend
сам подберёт следующий (`0.1.0` для первой draft'а либо bump от последней
published, см. handoff E4 §1.4 / E5 §1.2).

`transition` принимает `expected_status` — фронтенд всегда шлёт текущий статус
версии, чтобы получить 409 при concurrent transition'е (UI обновится через
invalidate, пользователь увидит свежий статус).

### 1.5. UI: `CodeSetPage`

- Card с версиями получил `extra` — кнопку «Создать DRAFT».
- Кнопка `disabled` если в `versions.data` найдена версия со статусом из
  `OPEN_STATUSES = {DRAFT, IN_REVIEW, STEWARD_APPROVED, OWNER_APPROVED}`. Это
  ровно at-most-one-open инвариант из E5 §1.2 — если попытаться создать вторую
  открытую, backend ответит 409, но UI не даёт даже до этого дойти.
- `Tooltip` сообщает, какая версия блокирует (полезно когда draft создал кто-то
  другой и текущий пользователь не понимает, почему кнопка серая).
- Открытая версия в списке помечена тегом «открытая» (i18n: `codeset.openVersion`).

### 1.6. UI: `WorkflowActions`

Матрица переходов (синхронизирована с backend `StateMachine` — handoff E5 §1.3 /
E6 §1.6):

| текущий статус | доступные кнопки | нужен комментарий |
|---|---|---|
| `DRAFT` | submit | — |
| `IN_REVIEW` | steward_approve, steward_reject | reject — да |
| `STEWARD_APPROVED` | owner_approve, owner_reject | reject — да |
| `OWNER_APPROVED`/`PUBLISHED`/`DEPRECATED`/`REJECTED` | (нет кнопок) | — |

`OWNER_APPROVED` намеренно не показывает publish — backend сразу триггерит
`PublishingService.autoPublish` (E6 §1.5). UI после `owner_approve` invalidate'ит
`getVersion`, и пользователь увидит уже `PUBLISHED`.

Reject через AntD `Modal` с `Form.Item` rules:
- `{ required: true }`,
- `{ whitespace: true }` — чистые пробелы не считаются,
- `{ min: 3 }` — защита от случайных «.», «??».

После успеха `setPending(null)` + `form.resetFields()`.

Frontend-side роли НЕ проверяются — backend gate'ит и через `@RolesAllowed`, и
через `OwnershipPort.rolesOf()` (handoff E5 §1.6). UI просто отдаёт кнопки;
если у пользователя нет прав — приходит 403, показывается через `message.error`.
**Это сознательно**: кнопок 5, состояний 7, ролей 4+ — frontend-валидация
дублировала бы backend и быстро рассинхронизировалась.

### 1.7. UI: `AddItemButton` + `AddItemModal`

Минимальный AntD Form:

| Поле | Парсинг | Валидация |
|---|---|---|
| `key_parts` (`Input`) | single-string `S1` → `["S1"]`; `[…]` → `JSON.parse` | required |
| `label_ru`, `label_en` (`Input`) | trim + null если пусто | maxLength 500 |
| `attributes` (`Input.TextArea`) | `JSON.parse` объект; пусто → `null` | — (валидация на backend'е по active CodeSetSchema) |
| `order_index` (`InputNumber`) | as-is | min 0 |

Парсинг key_parts/attributes — внутри `mutationFn`, чтобы `SyntaxError` попал в
`onError`. SyntaxError → показываем `<Alert>` под формой («Ошибка парсинга:
Unexpected token …»), модал не закрывается, пользователь правит и повторяет.

422 от backend'а (валидация против CodeSetSchema провалена) → `message.error` с
текстом ошибок (ApiError.message — формат `$.path: <message>`, см. handoff E4 §1.6).

После успеха — invalidate `["versions", versionId, "items"]` (все pages),
`qk.versions.one(vid)` (item_count), `qk.versions.byCodeset(codesetId)` (тоже
item_count в карточке). Модал закрывается, форма ресетится.

### 1.8. UI: `DeleteDraftButton`

`Popconfirm` поверх кнопки → `DELETE /versions/{id}`. После успеха —
`navigate('/codesets/{codesetId}', { replace: true })` + invalidate
`byCodeset` и `codesets.one`. `replace: true` — чтобы кнопка «Назад» не
вернула пользователя на удалённую страницу.

### 1.9. AntD App contextholder

Все mutation hooks используют `App.useApp().message` — это требует обёртки в
`<AntApp>` для корректного токена темы. В `main.tsx` это уже сделано (round 1):

```tsx
<ConfigProvider locale={...} theme={...}>
  <AntApp>{children}</AntApp>
</ConfigProvider>
```

Если использовать статический `message.error(...)` (импорт из `antd`), AntD 5
кричит warning в console про missing context — поэтому везде через `useApp()`.

### 2.2. Mutations (новое в round 2a)

| Endpoint | Метод | Используется в | Auth |
|---|---|---|---|
| `/api/v1/versions/by-codeset/{id}` | POST | CodeSetPage (Create DRAFT) | RDM_AUTHOR/RDM_ADMIN |
| `/api/v1/versions/{id}` | DELETE | VersionPage (Delete DRAFT) | RDM_AUTHOR/RDM_ADMIN |
| `/api/v1/versions/{id}/items` | POST | VersionPage (Add Item) | RDM_AUTHOR/RDM_ADMIN |

Семантика ошибок:
- 401 → `ApiError(401)` → `message.error` («401: Unauthorized»). Токен мог
  истечь — пользователь должен relogin (round 1 не реализует автоматический
  redirect на login при 401, см. round 1 follow-up §3 #4 в [`E11-ui.md`](E11-ui.md)).
- 403 → доступ запрещён на role-level либо asset-level (см. handoff E5 §2.2).
  `message.error` с конкретным сообщением от backend'а.
- 409 → state-machine отказ либо at-most-one-open. UI показывает текст ошибки
  через `message.error` и invalidate'ит relevant queries — пользователь увидит
  актуальное состояние.
- 422 → валидация payload'а (например, attributes против CodeSetSchema). UI
  показывает текст ошибок.

## 3. Что осталось доделать в E11.2a — мягкие follow-up'ы

Ничего не блокирует round 2b. Список того, к чему стоит вернуться:

1. **TanStack Query Devtools.** Не подключены. Добавить
   `<ReactQueryDevtools initialIsOpen={false}/>` в `App.tsx` под условием
   `import.meta.env.DEV` — упрощает дебаг кэшей в round 2b при появлении более
   сложных invalidation'ов.

2. **Optimistic updates пока не используются.** Все mutations — pessimistic
   (`mutate → onSuccess → invalidate → refetch → render`). Для кнопок workflow
   это норма (важно увидеть точный backend-ответ). Для PATCH ячеек grid'а в
   round 2b — стоит делать optimistic, иначе grid будет «дёргаться» на каждой
   правке.

3. **Bundle 1.35 МБ JS / 419 KB gzip.** Добавление `@tanstack/react-query` дало
   ~150 KB JS (~50 KB gzip). По-прежнему один chunk без code-splitting; см.
   [`E11-ui.md`](E11-ui.md) §3 #1. На round 2b или перед prod-deploy — добавить
   `manualChunks` в `vite.config.ts`.

4. **`AddItemModal` — composite key UX слабый.** Для одиночных ключей удобно
   (просто ввести строку), для composite key пользователь должен вручную писать
   JSON-array. В round 2b, когда подключим TanStack Table для inline-edit'а,
   стоит сделать composite-key через несколько Input'ов на части ключа,
   считывая `key_spec.parts` из CodeSet.

5. **`AddItemModal` — нет валидации attributes на frontend'е.** Все ошибки
   приходят с backend'а 422. Желательно подключить `ajv` (лёгкий JSON Schema
   валидатор) и валидировать против active CodeSetSchema **ДО** отправки —
   быстрее, без round-trip'а. Round 2b или V1+.

6. **Concurrent transitions: один пользователь может кликнуть Submit дважды
   очень быстро.** Кнопка `loading={transition.isPending}` блокирует второй
   клик — но только пока mutation в полёте. После завершения первой mutation'а
   до invalidate'а query пришёл refetch'нутого статуса есть микро-окно. На
   практике не воспроизводится без artifical lag'а. Если потребуется
   bullet-proof — можно использовать `useMutationState` + единый
   `mutationKey`-lock. Не critical.

7. **На странице нет кнопки "Refresh" / pull-to-refresh.** Если backend
   меняет состояние без вмешательства этой UI-сессии (например, другой
   пользователь сделал owner_approve), текущий пользователь увидит изменения
   только при следующем клике или через 30s staleTime. AntD-button + manual
   `queryClient.invalidateQueries(qk.versions.one(vid))` — тривиальное
   улучшение, не делал в этом round'е.

8. **`Space.Compact` в AddItemModal — AntD предупреждает про deprecated в 5.x.**
   Замена на `Flex` или `Row+Col`. Косметика.

9. **Старый `[]` deps в `useApi` мог отличаться от старого поведения.** В round
   1 при `[]` каждый mount был свежим запросом. С TanStack Query при
   `queryKey=qk.tasks.my()` запрос кэшируется глобально на 30s. Это **намеренное**
   улучшение (меньше нагрузка на backend), но если для какой-то страницы это
   критично (например, audit viewer должен всегда показывать самое свежее),
   можно явно `staleTime: 0` либо `refetchOnMount: 'always'` в опциях
   `useQuery`. Сейчас все 8 read-вызовов согласны на 30s cache.

  transition / at-most-one-open) → `message.error` с текстом ошибки от backend'а.
- `403` → role gate, `message.error`.
- `422` от валидации attributes — `message.error` с list of schema errors.
- `401` → `message.error("401: Unauthorized")`. Round 2a НЕ реализует
  автоматический redirect на login (см. round 1 follow-up §3 #4 в
  [`E11-ui.md`](E11-ui.md)).
