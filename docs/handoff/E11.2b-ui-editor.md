## 0. TL;DR за 30 секунд

- **Inline editor** на `ItemsTable` (включается prop'ом `editable`, активен только
  для DRAFT-версий). Колонка Actions с Edit/Save/Cancel/Delete. Edit-режим
  показывает inputs в ячейках label_ru/label_en/orderIndex и единый textarea
  «attributes (JSON)» (вместо разнесённых attr-колонок). PATCH идёт с
  `expected_row_version`; 409 → message + invalidate + откат UI; 422 → message
  с описанием schema-error.
- **Bulk import modal** (одна кнопка «Bulk-импорт» в Card extra на DRAFT):
  AntD `Tabs` JSON / CSV, каждый таб — drag-and-drop файла + textarea для
  paste. APPLIED → message с `+added/*updated/=unchanged`, invalidate items.
  REJECTED (HTTP 422) → in-modal `<Alert>` + таблица `errors[]` с
  `row_index/key_parts/field/message` — пользователь правит и повторяет
  без перезагрузки.
- **Diff viewer** (Drawer 80% width, кнопка «Сравнить с другой версией» доступна
  во всех статусах). Selector from-версии (отсортирован published_at DESC,
  current версия исключена) → ленивый `GET /diff?from=...` → summary statistics
  (added/changed/removed/moved цветными тегами) + таблица entries с
  `expand`-row'ом для side-by-side `before/after` JSON-pane'ов.
- **TanStack Query Devtools** подключены под `import.meta.env.DEV` —
  bottom-left button, не попадёт в prod-bundle (Vite tree-shake'ит ветку
  `import.meta.env.DEV` в production).
- **Code-splitting через `manualChunks`**: `vendor-react` (162 KB /
  53 KB gzip), `vendor-antd` (1 МБ / 325 KB gzip), `vendor-query`
  (171 KB / 48 KB gzip), `index` (62 KB / **19.9 KB gzip**). Initial
  parallel load — 4 запроса; при релизе app-кода вендоры не перевыкачиваются.

### 1.2. Inline editor — UX модель

**Состояние редактирования** живёт локально в `ItemsTable`: `editingId | null` +
один общий AntD `Form`. Только одна строка в edit-режиме одновременно — это
исключает риск «забыть сохранить»: при попытке отредактировать другую row
кнопка Edit неактивна, пока пользователь явно не Save или Cancel.

**Что можно править:**
- `label_ru`, `label_en` (Input)
- `order_index` (InputNumber)
- `attributes` — единая textarea с JSON-объектом (вместо разнесённых колонок
  по ключам, чтобы пользователь видел и мог менять любые поля сразу). После
  Save backend валидирует против active CodeSetSchema.

**Что НЕ редактируется через inline:**
- `key_parts` (immutable identity row'а — менять = удалять/создавать)
- `parent_key` / `parent_ref` (иерархия — отложено на E13 tree-editor)
- `effective_*` / `system_*` (bitemporal UI — E13)
- `status` (ACTIVE/RETIRED через отдельную операцию, V1+)

**Optimistic-lock UI:**

```
Edit row → Save → PATCH { expected_row_version: row.row_version, ... }
   ├─ 200 (success): message «Запись обновлена» + invalidate items + setEditingId(null)
   ├─ 409 (OptimisticLock OR DRAFT-only check):
   │     message «Запись была изменена другим пользователем. Данные обновлены — повторите.»
   │     + invalidate items (подтянем свежий row_version)
   │     + setEditingId(null) (откат UI: пользователь увидит свежие данные)
   └─ 422 (schema validation): message с описанием schema-error
```

**Rationale выбора UI-подхода**: handoff E11.2 §5 предлагал TanStack Table для
in-row edit'а, но AntD's editable cells работают через простой
`Form + isEditing(rowId)` switch — меньше boilerplate, копи-паст из Excel
поддерживается стандартным `<Input>` (Ctrl-V вставит в активное поле).
TanStack Table останется кандидатом на копи-паст-всей-строки или
multi-cell-edit (V1+).

### 1.3. Bulk import modal

Один компонент `BulkImportButton` + lazy `BulkImportModal`. AntD `Tabs` с
двумя табами:

| Таб | Backend endpoint | Парсинг |
|---|---|---|
| JSON | `POST /versions/{id}/items/bulk` | `JSON.parse` → `Array<NewItem>` + лёгкая sanity-проверка (key_parts обязателен; одиночная строка → массив длины 1) |
| CSV | `POST /versions/{id}/items/bulk-csv` (`Content-Type: text/csv`) | передаётся as-is (raw text), backend парсит через `jackson-csv` |

Каждый таб содержит:
- `Upload.Dragger` — drag-and-drop файла (`.json` или `.csv`); `beforeUpload`
  возвращает `false` (мы только читаем содержимое в textarea, default upload
  не вызываем).
- `<textarea>` для paste'а вручную (с placeholder'ом-примером).

**Outcome handling:**
- `200 + status=APPLIED` → `message.success` с цифрами + invalidate +
  `result` отображается в `<ResultPanel>` (Statistic + успех).
- `422 + status=REJECTED` → `ApiError(422, body=BulkImportResult)`. Поскольку
  тело ошибки — это ровно тот же `BulkImportResult`, в onError проверяем
  `body.status === "REJECTED"` и показываем тот же `<ResultPanel>` с таблицей
  `errors[]` (`row_index/key_parts/field/message`). **Модал не закрывается** —
  пользователь правит payload и нажимает «Загрузить» ещё раз.
- `SyntaxError` (битый JSON в JSON-табе) → in-modal `<Alert>` («Ошибка
  парсинга payload'а: …»), без обращения к backend'у.

### 1.4. Diff viewer

`DiffButton` доступен на любой версии (даже DRAFT — полезно посмотреть, что
изменилось от последней published). Drawer открывается на 80% ширины. Внутри:

1. **Selector from-версии** — берётся из уже-загруженного
   `listVersionsByCodeSet(codesetId)` (см. `VersionPage` §1.5 ниже),
   фильтруется (исключаем current `toVersion.id`), сортируется
   `published_at DESC ?? created_at DESC`.
2. **Lazy-fetch** — `useQuery` с key `qk.versions.diff(toId, fromId)`
   запускается только когда `fromId !== null`. Backend `GET /diff?from=...`.
3. **Summary statistics** — 6 `<Statistic>` строк: from / to / ADDED /
   CHANGED / REMOVED / MOVED (цветные теги).
4. **Таблица entries** — колонки op (filterable Tag), key_parts (моноширинно),
   changed_fields (Tags). Каждая row expandable → `<BeforeAfter>` (две
   pre-колонки для JSON before/after).

`Drawer destroyOnClose` — при закрытии from-выбор сбрасывается. Diff
кэшируется QueryClient'ом (staleTime 30s) — повторный открытие на той же
паре (toId, fromId) мгновенно из cache.

### 1.5. VersionPage — изменения

- Добавлен load `codesetVersions` (для DiffViewer'а) — по
  `qk.versions.byCodeset(codesetId)`. Если `codesetId` ещё не загружен,
  query-key содержит "_pending" placeholder, query вернёт `[]` без HTTP-запроса
  (loader с условным `Promise.resolve([])`).
- В Card «Записи» extra — теперь **две** кнопки в `<Space>`: «Bulk-импорт» +
  «Добавить запись» (обе только в DRAFT).
- В блоке actions под Descriptions — добавлена кнопка `<DiffButton>` (всегда
  видна, если codesetVersions загружены).
- ItemsTable получает `editable={status==="DRAFT"}` + versionId + codesetId
  для wire-up'а mutation invalidate'ов.

### 1.6. Code-splitting

`vite.config.ts:build.rollupOptions.output.manualChunks`:

| Chunk | Содержимое | Размер |
|---|---|---|
| `vendor-react` | react, react-dom, react-router-dom | 162 KB / **53 KB gzip** |
| `vendor-antd` | antd, @ant-design/icons | 1 МБ / **325 KB gzip** |
| `vendor-query` | @tanstack/react-query, @tanstack/react-table, oidc-client-ts, i18next + plugins | 171 KB / **48 KB gzip** |
| `index` (app) | весь src/ кроме вендоров | 62 KB / **19.9 KB gzip** |

Initial parallel load — 4 HTTP-запроса. При релизе нового кода приложения
вендоры не перевыкачиваются (browser cache попадает по контентному hash'у в
имени файла).

`vendor-antd` всё ещё большой — это **acceptable**, потому что:
- кэшируется отдельно от app-кода;
- AntD не tree-shake'ится без `babel-plugin-import` / `vite-plugin-imp`
  (только partial — отдельные subcomponents подтягиваются полностью);
- в долгосрочной перспективе альтернативы две: (a) написать
  `vite-plugin-imp` setup для AntD 5; (b) выпиливать редко-используемые
  AntD-компоненты. Обе — оптимизация V1+ или round 2c.

### 1.7. TanStack Query Devtools

Подключены в `App.tsx` под `import.meta.env.DEV` — production-бандл их
исключает (Vite tree-shake'ит ветку под `if (import.meta.env.DEV)`,

закрыта.

Полезно для дебага invalidate-цепочек (например, после Save row →
проверить, что `qk.versions.itemsRoot(vid)` получил `invalidate`).

### 2.2. Mutations (новое в round 2b)

| Endpoint | Метод | Используется в | Auth |
|---|---|---|---|
| `/api/v1/versions/{id}/items/{itemId}` | PATCH | ItemsTable (Edit row) | RDM_AUTHOR/RDM_ADMIN |
| `/api/v1/versions/{id}/items/{itemId}` | DELETE | ItemsTable (Delete row) | RDM_AUTHOR/RDM_ADMIN |
| `/api/v1/versions/{id}/items/bulk` | POST | BulkImportModal (JSON tab) | RDM_AUTHOR/RDM_ADMIN |
| `/api/v1/versions/{id}/items/bulk-csv` | POST | BulkImportModal (CSV tab) | RDM_AUTHOR/RDM_ADMIN |
| `/api/v1/versions/{id}/diff?from={fromId}` | GET | DiffViewer | любая authenticated |

Семантика ошибок (in addition to round 2a §2.2):
- `409` PATCH → optimistic-lock conflict либо DRAFT-only check провалился
  (status больше не DRAFT). UI трактует одинаково: invalidate + откат.
- `422` PATCH → CodeSetSchema validation failed на attributes. UI: `message.error`.
- `422` bulk JSON/CSV → `BulkImportResult{status:"REJECTED", errors:[...]}` в
  теле ApiError. UI рендерит errors-таблицу в модале.
- `400` bulk JSON → отдельный path (например, payload не array). Сейчас не
  выделен в UI — приходит общим message.error.

## 3. Что осталось доделать в E11.2b — мягкие follow-up'ы

Ничего не блокирует round 2c. Список того, к чему стоит вернуться:

1. **Composite-key UX в AddItemModal** (debt из E11.2a §3 #4 не закрыт). При
   composite ключе пользователь по-прежнему вводит JSON-array строкой. План:
   получить `key_spec.parts` из `getCodeSet(...)`, рендерить N отдельных
   Input'ов по `parts[].name`, тип Input — по `parts[].type` (STRING → Input,
   INTEGER → InputNumber, BOOLEAN → Switch, DATE → DatePicker). Round 2c
   либо отдельный мини-PR.

2. **Frontend-валидация attributes через `ajv`** (debt из E11.2a §3 #5). Сейчас
   все валидационные ошибки приходят с backend'а 422. С `ajv` можно валидировать
   ДО отправки и показывать ошибки прямо в форме без round-trip'а. Маленький
   плюс к UX, не critical. Полезно особенно в Bulk import (вычислять
   `errors[]` локально перед POST'ом).

3. **AntD chunk 325 KB gzip остаётся большим.** Не блокирует пилот (vendored,
   кэшируется). Когда приоритеты дойдут — добавить `vite-plugin-imp` или
   ESM-import'ы только нужных subcomponents (например, `import { Button } from
   "antd/es/button"`). Цель — initial parallel load < 200 KB total gzip.

4. **Inline-edit не поддерживает многострочное редактирование.** Если пользователь
   правит одну row, кликает Edit на другую — кнопка серая. Excel-style
   tab/enter navigation между ячейками — TanStack Table даёт это из коробки,
   AntD editable нет. V1+ либо переключиться на TanStack для grid-edit-tier'а.

5. **Bulk import — нет валидации заголовков CSV.** Backend валидирует и при
   ошибке возвращает 422 с описанием. Можно подсмотреть формат через
   `csv-parse` / `papaparse` на frontend'е и показать ошибки заранее.

6. **Bulk import — нет preview.** Пользователь paste'ит payload, кликает
   «Загрузить», только после backend-round-trip'а видит результат. Можно
   добавить кнопку «Preview» — локальный JSON.parse + table-render пары
   первых строк. V1+.

7. **Diff `before/after` JSON-pane не подсвечивает изменённые поля.** Сейчас
   просто `JSON.stringify(before, null, 2)` vs `after`. С json-diff-libу
   (`diff-match-patch` или `react-diff-viewer`) можно получить inline-highlight.
   Удобно при `changed_fields=["attributes.pd"]` на матрицах PD/LGD.

8. **Drawer destroyOnClose сбрасывает выбранную from-версию.** Это
   намеренно — каждое открытие диалога начинается с чистого выбора. Если
   пользователю нужно сравнивать одну и ту же пару много раз — пере-выбирать
   from. Cache в QueryClient переживает закрытие, поэтому повторный fetch
   мгновенный.

9. **`BulkImportButton` рендерится даже когда `code_set_schema` ещё не
   загрузился.** На пустом codeset'е без атрибутов это ОК (CSV/JSON примут
   любой payload, упадёт 422 с понятным сообщением). Но было бы хорошо
   подгружать active schema и показывать её JSON как hint в hover-tooltip'е.

10. **`vendor-query` chunk включает `oidc-client-ts`** (он логически
    auth-стек, не query-стек), потому что иначе попадает в `index`-chunk и
    раздувает app-код. Логически разнести можно через ещё один `vendor-auth`
    chunk — micro-cleanup, не критично.

## 4. Технический долг и решения, повлиявшие на следующие раунды

| Что | Где | Когда снять / следующий шаг |
|---|---|---|
| AntD inline-edit вместо TanStack Table | `ItemsTable` | Если потребуется multi-cell-edit / Excel-tab-navigation — переезд на TanStack Table. Сейчас 1-row-edit'а достаточно. |
| `attributes` редактируется как единая JSON-textarea | `ItemsTable` | Потенциально расщепить на отдельные Input'ы по схеме (как в `AddItemModal`-debt §3 #1). Но JSON удобнее, когда атрибутов много или вложенные. Решение оставить — не снимать без явного запроса UX. |
| `key_parts`/`parent_key`/`effective_*` в edit не редактируются | `ItemsTable` | Move-операция (parent_key) и bitemporal (effective_*) — отдельный UI в E13. Изменение key_parts — это пересоздание row'а, не PATCH. |
| AntD chunk 325 KB gzip | `vite.config.ts` | V1+: `vite-plugin-imp` или narrow-imports. |
| `vendor-query` объединяет query + table + oidc + i18n | `vite.config.ts` | Возможно разнести на `vendor-query` + `vendor-auth` + `vendor-i18n` — но `index` сейчас 19.9 KB gzip, без острой необходимости. |
| Devtools-зависимость в `package.json` как devDep | `package.json` | Корректно, не попадает в prod. |
| `qk.versions.byCodeset(null ?? "_pending")` placeholder pattern | `VersionPage` | Чтобы избежать `enabled: !!codesetId` ветки — упрощает код. Минор: первый рендер делает один лишний `Promise.resolve([])`. Не оптимизировать без реального impact'а. |
| Bulk REJECTED → `result` отображается из `ApiError.body` | `BulkImportModal` | Хрупко: если backend изменит формат ошибки — UI не покажет details. Есть fallback на общий `message.error`. Стабильно через rdmmesh-spec/schema/api/bulk-import-result.json. |

### 6.2. End-to-end (требует `make up` от backend'а; **не прогонялся в этой сессии**)

```bash
# 1. backend
make up

# 2. UI
make ui-install   # один раз
make ui

# 3. в браузере (предполагаем DRAFT-версию V с минимум 1 item):
#   /versions/{V} как dev-author:
#
#   ИНЛАЙНОВЫЙ EDIT:
#   - в Card "Записи" → колонка "Действия" → Edit на S1
#   - меняем label_ru="Стадия 1 (изм.)" + attributes={"stage":"1","note":"upd"}
#   - Save → message «Запись обновлена», row_version: 0 → 1
#   - повторно Save с тем же body — backend не должен ругаться (idempotent)
#   - в БД: docker exec rdmmesh-postgres psql -c \
#       "SELECT row_version, label_ru FROM authoring.code_item WHERE id='<S1-id>';"
#
#   OPTIMISTIC LOCK:
#   - в одной вкладке открыть Edit S2; не сохранять
#   - в другой вкладке Edit S2 → Save (row_version 0 → 1)
#   - первая вкладка → Save → 409 → message «Запись была изменена другим
#     пользователем. Данные обновлены — повторите.» + UI обновился
#
#   DELETE:
#   - Delete S3 → Popconfirm → OK → message «Запись удалена», item_count 3 → 2
#
#   BULK JSON:
#   - кнопка «Bulk-импорт» → таб JSON → paste:
#       [{"key_parts":["X1"],"label_ru":"Икс 1","attributes":{"stage":"1"}}]
#   - Загрузить → message «Применено: +1/*0/=0», invalidate, новая row в grid'е
#   - paste с {"stage":"4"} → 422 → in-modal таблица errors[],
#     row_index=0, message содержит "$.attributes.stage: …"
#
#   BULK CSV:
#   - таб CSV → paste:
#       key_parts,label_ru,label_en,attributes
#       Y1,Игр 1,Y 1,"{""stage"":""1""}"
#       Y2,Игр 2,Y 2,"{""stage"":""2""}"
#   - Загрузить → APPLIED +2
#
#   DIFF:
#   - на любой версии (DRAFT работает тоже) → кнопка «Сравнить с другой
#     версией» → Drawer → Select from-версии → загружается diff
#   - summary: цифры по op'ам (+ цветные теги)
#   - row entry → expand → before/after pane (JSON-pretty)
#   - filter по op (стандартный AntD column filter)
#
#   Devtools:
#   - bottom-left button (кружок) → панель TanStack Query Devtools
#   - после Save row видно invalidate'ы по qk.versions.itemsRoot(vid),
#     qk.versions.one(vid), qk.versions.byCodeset(csid)
```

Ожидаемые ошибки и их UI-обработка:
- 409 на PATCH → message «Запись была изменена …» + invalidate.
- 422 на PATCH → message с описанием (формат `$.path: <message>`).
- 422 на bulk → in-modal `<ResultPanel>` с таблицей errors.
- 400 на bulk JSON (не array) → message.error общий.
- 404 на DELETE (item уже удалён) → message.error общий.
- 401 на любом → message «401: Unauthorized». Round 2c добавит
  автоматический redirect на /login (см. round 1 follow-up §3 #4 в
  [`E11-ui.md`](E11-ui.md)).
