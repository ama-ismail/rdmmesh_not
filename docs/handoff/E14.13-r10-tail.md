## 0. TL;DR

1. **Pin тега.** `rustfs/rustfs:latest` → `rustfs/rustfs:1.0.0-beta.3`
   (новейший конкретный, проверен по Docker Hub). Override —
   `RDM_RUSTFS_IMAGE`. Воспроизводимость dev/CI.
2. **Q62 — Object-Lock mode конфигурируем.** `RDM_ARCHIVE_LOCK_MODE`
   (`GOVERNANCE`|`COMPLIANCE`), **дефолт GOVERNANCE** (operationally
   safe — bypass-роль может исправить ошибку). `COMPLIANCE` —
   regulator-mandated (IFRS9 неизменяемость, необратимо до retain-until
   даже для root). Невалидное значение → GOVERNANCE + error-лог.
3. **Cron-архивация + runbook.** `docs/runbooks/audit-archive.md`:
   ежемесячный k8s CronJob (token client_credentials → `POST
   /audit/archive` прошлого месяца → `verify` → guarded drop вне
   retention), идемпотентность, failure-матрица, Q62-гайд. Решение по
   планировщику — **ops/runbook + k8s CronJob**, НЕ embedded-scheduler
   (консистентно с E14.7 partition-cron; SPEC §3.1 lean — без лишней
   in-process инфраструктуры). `audit-log-retention.md` §3 обновлён:
   штатный дроп — 1-арг V074 (honor-system устранён), ссылка на новый
   runbook.

Backend-конфиг + docs. Кода мало (адаптер + фабрика), миграций нет.

## 2. Решения

- **Дефолт GOVERNANCE, не COMPLIANCE.** COMPLIANCE необратим — ошибочно
  заархивированный объект залочен на 10 лет без отмены даже для root.
  Дефолт безопаснее операционно; на COMPLIANCE флипать осознанно после
  проверки пайплайна (см. runbook §4). Режим — per-object на момент PUT;
  смена env не меняет ранее записанные объекты.
- **Cron = k8s CronJob, не embedded.** SPEC §3.1 (lean, 3 внешних
  компонента) + прецедент E14.7 (`ensure_audit_partition` — ops-задача).
  Embedded-scheduler потребовал бы синтетического RDM_ADMIN-principal'а
  (усложняет no-bypass-постуру) — отвергнуто.
- **Pin конкретного тега** вместо `latest` — воспроизводимость; бамп
  тега = осознанный шаг после регресс-smoke.

## 4. Smoke — COMPLIANCE vs реальный RustFS (2026-05-17)

```
./bin/mvn -DskipITs verify → BUILD SUCCESS, ArchUnit 11
docker compose build rdmmesh-service && down -v
RDM_ARCHIVE_LOCK_MODE=COMPLIANCE docker compose up -d
docker inspect rdmmesh-rustfs → rustfs/rustfs:1.0.0-beta.3   (pin работает)
лог: "ArchivePort enabled … lockMode=COMPLIANCE"             (env honored)
seed → POST /audit/archive?year=2026&month=5 → retentionApplied:true, rows 4
mc stat (независимо):
  X-Amz-Object-Lock-Mode             : COMPLIANCE
  X-Amz-Object-Lock-Retain-Until-Date: 2036-06-01T00:00:00Z   (month+10y)
mc retention info: Mode: COMPLIANCE, expiring in 3667 days
GET /audit/archive/audit_log_y2026m05/verify → verified:true, sha match
```

Подтверждено независимым S3-клиентом (не самоотчёт сервиса): объект
реально под **COMPLIANCE** Object-Lock. GOVERNANCE-путь — регресс не
затронут (тот же код, отличается только enum; verified ранее E14.11 §5).
