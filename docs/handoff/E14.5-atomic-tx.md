## 0. TL;DR за 30 секунд

- **Проблема.** `WorkflowService.transition` делал три действия в **разных**
  Postgres-транзакциях:
  1. `lifecycle.transition(...)` — CAS status + reviewer/approver UPDATE
     (authoring schema).
  2. `jdbi.useTransaction(...)` — INSERT в `workflow.workflow_transition` +
     UPSERT `workflow.approval_task`.
  3. `eventBus.publish(...)` — async подписчик audit INSERT'ит в `audit.audit_log`.

  Между шагами 1 и 2 был **зазор**: если шаг 2 падал (BD-сбой, OOM, network),
  status уже изменён, журнала нет — compliance-проблема.

- **Решение.**
  - **`VersionLifecyclePort.transition(Handle, ...)`** — новый overload.
    Принимает Handle от caller'а и работает на нём, не открывая собственную tx.
    Backward-compatible: старый overload без Handle делегирует на новый через
    свой `jdbi.inTransaction(...)`.
  - **`VersionLifecycleAdapter`** реализует оба overload'а; внутренняя логика
    (`casStatus` + `recordReviewer` + `setApprover`) теперь в одном
    `transition(Handle, ...)`.
  - **`WorkflowService.transition`** объединил шаги 1+2 в одну
    `jdbi.inTransaction(handle -> {...})`: lifecycle.transition(handle, ...) +
    `WorkflowTransitionDao.insert(...)` + `ApprovalTaskDao.closeAll/upsertOpen(...)`.
    Шаг 3 (`eventBus.publish`) остаётся ПОСЛЕ commit'а — это by-design, audit
    best-effort (SPEC §3.8).

- **`rdmmesh-api` pom.xml** теперь имеет dep на `org.jdbi:jdbi3-core`. JDBI —
  корпоративный transactional primitive у проекта; добавление его в api
  трактуется как trade-off ради atomic-tx primitive (см. §4).

- **Performance улучшение бонусом.** Кандидаты на следующую approval-task
  (`candidatesFor(...)` → `ownership.ownersOf(...)`) теперь pre-fetch'атся **до**
  входа в write-tx. Это сокращает время удержания row-lock'а на
  `code_set_version` и `approval_task` примерно на длительность ownership-read'а
  (~ms).

## 1. Что сделано

### 1.1. `VersionLifecyclePort` — Handle-overload

**Файл:** `rdmmesh-api/src/main/java/bank/rdmmesh/api/port/VersionLifecyclePort.java`

```java
import org.jdbi.v3.core.Handle;

public interface VersionLifecyclePort {
    /** ... старый overload — backward-compatible, открывает собственную tx ... */
    boolean transition(UUID versionId, String fromStatus, String toStatus,
                       UUID actor, TransitionEffect effect);

    /**
     * E14 round 5: atomic-decision overload. Принимает чужой {@link Handle} и
     * выполняет CAS + side-effects на нём — без открытия собственной транзакции.
     * Это позволяет вызывающему коду (например, {@code WorkflowService.transition})
     * объединить authoring-update (status CAS) и workflow-INSERT'ы в одну
     * Postgres транзакцию через {@code Jdbi.inTransaction}.
     */
    boolean transition(Handle handle, UUID versionId, String fromStatus,
                       String toStatus, UUID actor, TransitionEffect effect);
    ...
}
```

### 1.2. `VersionLifecycleAdapter` — реализация обоих overload'ов

```java
@Override
public boolean transition(UUID versionId, ...) {
    // Backward-compatible: открываем свою tx и делегируем.
    return jdbi.inTransaction(handle ->
            transition(handle, versionId, fromStatus, toStatus, actor, effect));
}

@Override
public boolean transition(Handle handle, UUID versionId, ...) {
    // Чужой handle — работаем в его tx.
    CodeSetVersionDao dao = handle.attach(CodeSetVersionDao.class);
    int n = dao.casStatus(versionId, fromStatus, toStatus);
    if (n == 0) return false;
    if (effect.recordReviewer()) dao.recordReviewer(versionId, actor);
    if (effect.setApprover())    dao.setApprover(versionId, actor);
    return true;
}
```

Legacy-callers (тесты, любой код вне WorkflowService) работают как раньше.
Новый callsite — WorkflowService — берёт overload с Handle.

### 1.3. `WorkflowService.transition` — одна tx

**Файл:** `rdmmesh-workflow/src/main/java/bank/rdmmesh/workflow/internal/service/WorkflowService.java`

```java
TransitionEffect effect = new TransitionEffect(
        decision.recordReviewer(), decision.setApprover());

// Кандидаты на следующую approval-task — pre-fetch ДО tx, чтобы не держать
// ownership-read внутри write-tx (длительность ownership lookup + row-lock
// на code_set_version → лишний contention).
String nextRole = StateMachine.nextRequiredRole(to);
UUID[] candidates = (nextRole == null)
        ? null
        : candidatesFor(version.codesetId(), nextRole);

UUID transitionId = UUID.randomUUID();
Instant occurredAt = Instant.now();

// E14 round 5: ОДНА tx на CAS + journal + approval-task.
boolean applied = jdbi.inTransaction(handle -> {
    boolean ok = lifecycle.transition(
            handle, versionId, version.status(), targetStatus, actor, effect);
    if (!ok) return false;

    handle.attach(WorkflowTransitionDao.class).insert(
            transitionId, versionId, version.codesetId(), codeSet.domainId(),
            version.status(), targetStatus, decision.action().name(), actor, comment);

    ApprovalTaskDao tasks = handle.attach(ApprovalTaskDao.class);
    tasks.closeAll(versionId, actor);
    if (nextRole != null) {
        tasks.upsertOpen(versionId, version.codesetId(), codeSet.domainId(),
                         nextRole, candidates);
    }
    return true;
});

if (!applied) {
    throw new IllegalStateTransitionException(
            "Конкурентный transition: версия " + versionId + " ...");
}

// EventBus.publish ПОСЛЕ commit'а — audit/publishing subscribers видят
// зафиксированное состояние, а не in-flight tx. Best-effort by design.
try {
    eventBus.publish(new WorkflowTransitionDomainEvent(...));
} catch (RuntimeException e) {
    log.warn("workflow: event publish failed (transition_id={}): {}",
             transitionId, e.toString());
}
```

**Поведение при CAS fail:**
- `lifecycle.transition(handle, ...)` возвращает `false` если status уже
  изменён другим actor'ом.
- Лямбда возвращает `false` → tx commit'ится без changes (или rollback'ается;
  Jdbi одинаково обрабатывает оба случая, потому что мы не writeали).
- `applied=false` → бросаем `IllegalStateTransitionException`.

**Поведение при сбое внутри tx (после успешного CAS):**
- Любое RuntimeException из `WorkflowTransitionDao.insert(...)` или
  `ApprovalTaskDao.upsertOpen(...)` → `jdbi.inTransaction` rollback'ает всё:
  status возвращается на исходный, journal-row не остаётся.
- Caller получает `JdbiException` → Resource layer мапит в 5xx.
- Status и журнал **остаются консистентны**.

### 1.4. `rdmmesh-api/pom.xml` — JDBI как dep

```xml
<!-- E14 round 5: VersionLifecyclePort.transition имеет overload, принимающий
     org.jdbi.v3.core.Handle для atomic shared-tx ... -->
<dependency>
    <groupId>org.jdbi</groupId>
    <artifactId>jdbi3-core</artifactId>
</dependency>
```

**Trade-off.** Раньше `rdmmesh-api` имел deps только на `rdmmesh-spec` и
`jakarta.validation`. JDBI — это конкретная library; формально api теряет
agnosticism. Альтернативы:

| Подход | Pro | Con |
|---|---|---|
| **JDBI dep в api** (выбран) | Минимум кода. Use site читается естественно. | api привязан к JDBI. |
| TxHandle-абстракция в api | api нейтральный. | Дополнительный wrapper class в каждом adapter'е. Один use-case на пилоте — overkill. |
| Pure callback в port (без Handle) | api нейтральный. | Callback должен иметь типизированный receiver — снова требует Handle. |
| Перенести transition-логику в WorkflowService | Одна tx через прямые DAO calls. | Нарушает SPEC §3.3: schema authoring пишет ТОЛЬКО authoring модуль. |

JDBI — это de-facto единственный transactional primitive проекта; **все**
adapter'ы используют его. Эквивалентно тому, как `javax.transaction.UserTransaction`
или Spring `TransactionStatus` неявно «протекают» в API контракты больших
enterprise-проектов. Решение задокументировано в pom.xml-комментарии.

### 1.5. Performance: pre-fetch candidates

Раньше `candidatesFor(version.codesetId(), nextRole)` вызывался **внутри**
`jdbi.useTransaction(...)` — ownership lookup делал свою jdbi.withExtension
read-tx (nested savepoint в Postgres), удерживая основной write-tx ещё на
~ms.

Теперь `candidatesFor` — pre-fetch до tx. Семантически безопасно: ownership
не меняется во время transition (OM webhook идёт в свою tx; race окно
ничтожно мало по сравнению с длительностью самого 4-eyes flow).

### 2.1. Backend без изменений

REST API — без изменений. `POST /api/v1/versions/{id}/transitions` принимает

Любой реализатор `VersionLifecyclePort` (для тестов / mock'ов) должен
реализовать **оба** transition'а. На пилоте есть только
`VersionLifecycleAdapter`; mock'и можно делегировать new→old:

```java
@Override public boolean transition(Handle h, UUID id, ..., TransitionEffect e) {
    return transition(id, ..., e);  // delegate to legacy overload
}
```

### 2.3. Frontend

Не тронут. UX и observable behavior идентичны.

## 3. Что осталось — known split-tx (для следующих round'ов)

Round 5 закрыл самый частый flow (4-eyes transition). Остаются три split-tx,
которые менее частые но всё ещё compliance-relevant:

### 3.1. PublishingService.autoPublish — split на 3 tx

Текущее (handoff E6 §3 #1):
1. `lifecycle.publish(...)` — CAS OWNER_APPROVED → PUBLISHED + content_hash +
   signature (authoring tx).
2. `journal.recordSystemTransition(...)` — INSERT system-action в
   `workflow.workflow_transition` (workflow tx).
3. `outbound.enqueueVersionPublished(...)` — INSERT в
   `publishing.webhook_outbox` (publishing tx).

Возможные сценарии split-фейла:
- Step 1 OK, Step 2 fail → версия PUBLISHED, нет system-transition записи.
- Step 1+2 OK, Step 3 fail → версия PUBLISHED, outbox пуст, consumer'ы не
  получат webhook.

**План фикса:** аналогично round 5 — добавить Handle-overload'ы в
`OutboundPort.enqueueVersionPublished(Handle, ...)` и в JournalPort
(если есть отдельный port). PublishingService.autoPublish объединяет всё в
одну `jdbi.inTransaction(handle -> {...})`.

Сложность: M. Затрагивает 3 порта, 3 adapter'а. Сделать round 5.1 или
объединить с round 6 (HMAC rotation).

### 3.2. OwnershipWebhookService — split на 2 tx

Текущее (handoff E7 §3):
1. `processed_om_event` INSERT (ownership tx, idempotency).
2. `rdm_asset_ownership` UPSERT либо `catalog.domain` UPSERT (ownership tx).

Это уже **в той же схеме**, но в двух разных jdbi-call'ах. Может быть split,
если первая прошла, а вторая упала.

**План фикса:** простой — обернуть OwnershipWebhookService.handle(...) в
одну `jdbi.inTransaction(handle -> {...})`. Не требует port-overload'ов
(ownership module владеет обеими таблицами).

Сложность: S. Можно сделать как часть round 5.2 либо отдельным мини-PR'ом.

### 3.3. AuditService.onEvent — best-effort by design

Текущее (handoff E10 §3 #4):
- AuditService подписан на EventBus и пишет в `audit.audit_log` в **отдельной**
  tx после commit'а основной workflow/publishing tx.
- Если audit-INSERT упадёт — основная tx уже на диске, audit-row пропущен.

**Это by-design** (SPEC §3.8): append-failure не должен блокировать бизнес-
операцию. Audit — best-effort.

**Если потребуется true atomic audit** (например, regulator-mandated):
- Outbox-pattern: каждый publisher (workflow/publishing/ownership) пишет в
  `<module>.event_outbox` в той же tx что state-change. Background-flusher
  читает outbox и INSERT'ит в audit_log.
- Это разрушает текущую event-bus архитектуру и требует ~10× больше кода.
- **Не делать до явного compliance-mandata.**

## 4. Технический долг и решения

| Что | Где | Когда снять / следующий шаг |
|---|---|---|
| JDBI dep в `rdmmesh-api` | pom.xml | by-design на пилоте; рефакторим если api modules-count вырастет. |
| candidatesFor pre-fetch вне tx — race с ownership webhook'ом | WorkflowService | Race окно ms-level; ownership change через webhook идёт в свою tx; типовой кейс race не воспроизводится. Если попадётся в проде — добавить retry на CAS-fail (transient race). |
| `lifecycle.transition` old overload остался | Adapter | by-design — backward-compat для тестов и legacy-callers. Дешёво. |
| PublishingService split-tx (3 tx) | §3.1 | Round 5.1 либо round 6 bundle. |
| OwnershipWebhookService split (2 tx, same schema) | §3.2 | Round 5.2 либо отдельный mini-PR. |
| audit-INSERT в собственной tx | AuditService | by-design (audit best-effort, SPEC §3.8). |
| Тесты на atomic-rollback (force fail после CAS) | testcontainers IT | Round 9 — нужен реальный Postgres для force-throw + verify журнал не остался. |
| `transitionId/occurredAt` генерятся снаружи tx | WorkflowService | OK — `Instant.now()` снаружи tx даёт «момент initiation» сценария; transaction commit может быть на ms позже. Совпадает с прежним поведением (он тоже генерил их снаружи). |

## 6. Открытые вопросы

Без изменений с E14.4, плюс round 5 specific:

1–46 — без изменений (см. `E14.4-audit-export.md` §6).
47. **JDBI в `rdmmesh-api` — установлено и далее?** Сейчас один use-case
    (`VersionLifecyclePort.transition`). Если появятся другие atomic-tx
    primitives (например, `OutboundPort.enqueue(Handle, ...)` в round 5.1),
    JDBI в api станет нормой. Возможно стоит явно legitimize в ADR-0001
    («JDBI Handle — accepted transactional primitive in api»).
48. **Round 5.1+5.2: bundle или раздельно?** PublishingService split (3 tx)
    и OwnershipWebhookService split (2 tx) — независимые рефакторинги.
    Bundle короче по handoff-документации, но раздельные PR'ы безопаснее
    в review.
49. **Outbox-pattern для true atomic audit?** Compliance может потребовать
    "audit-row commit'ится atomically с state-change". Это разрушает
    текущую event-bus архитектуру; цена ~10× в коде. До явного mandata
    — не делать.
