## 0. TL;DR за 30 секунд

- **Новая страница** `/admin/subscriptions` — `SubscriptionsPage.tsx`. CRUD над
  outbound webhook subscriptions из E9 (REST `/api/v1/subscriptions`):
  - **Список** в AntD `<Table>`: id (truncated + tooltip), url (copyable
    ellipsis), secret_id (моноширинно), filter (цветные теги по domains/
    codesets/events), active/inactive Tag, last_delivery_status (цветной Badge
    + Tag + timestamp), created_at.
  - **Создать** — primary-кнопка → AntD `Modal+Form`: url (URL-validated),
    secret_id, filter (3 multiselect'а domains/codesets/events; пустые поля =
    нет ограничения по оси), active toggle (default ON).
  - **Деактивировать** — `Popconfirm` + DELETE (soft-delete; backend никогда не
    удаляет физически — FK на webhook_outbox + audit). Кнопка disabled на уже
    inactive строках.
- **Conditional Sider-меню** — `AppLayout.tsx` через `useAuth().baseRoles`:
  пользователи без `RDM_ADMIN` не видят раздел «Админ → Subscriptions».
  Frontend-guard внутри `SubscriptionsPage` отдаёт «Доступ запрещён» при
  попытке зайти по прямой ссылке. **Backend остаётся authoritative gate'ом**
  через `@RolesAllowed("RDM_ADMIN")` — frontend-проверка только UX.
- **Bundle:** app-код вырос с 19.9 → **22.6 KB gzip** (+SubscriptionsPage),
  vendor-antd подрос на ~6 KB gzip (подтянулись Switch/Badge), всё остальное
  без изменений.

└── i18n/{ru,en}.json                    ← +nav.{admin, subscriptions}, +subs.*

### 1.2. Routing + Sider

`App.tsx` добавил `<Route path="/admin/subscriptions" element={<SubscriptionsPage />} />`
под уже существующим `<ProtectedRoute>` (т.е. требует JWT, как и все остальные
страницы).

`AppLayout.tsx` — Sider-меню теперь композируется в зависимости от роли:

```
items = [
  { catalog },
  { tasks },
  ...(isAdmin ? [
    { type: "divider" },
    { type: "group", label: "Админ", children: [
      { subscriptions }
    ]}
  ] : []),
]
```

`isAdmin = baseRoles.includes("RDM_ADMIN")` — `baseRoles` приезжает из JWT
claim'а `groups` (handoff E2 §1.1 / E11 §1.3). При смене языка/relogin'а
list пересчитывается через `useMemo([t, isAdmin])`.

### 1.3. SubscriptionsPage

**Layout:**

```
<Card title="Outbound webhook subscriptions"
      extra={<Button "Создать subscription" />}>
  <SubscriptionsTable /> // или Loader/Alert при error
</Card>
<CreateSubscriptionModal open onClose />
```

**SubscriptionsTable** (AntD `<Table>` без пагинации; на пилоте subscriptions
будет 1-2 шт., таблица — flat):

| Колонка | Render |
|---|---|
| id | `<Tooltip title={id}><code>{id.slice(0,8)}…</code></Tooltip>` |
| url | `<Typography.Text copyable={{text:url}} ellipsis>` |
| secret_id | `<code>` |
| filter | `<FilterBadge>` — цветные теги: cyan для `d:risk`, geekblue для `c:ifrs9_stages`, purple для `VERSION_PUBLISHED`. Пустой фильтр → серый Tag «все события». |
| active | `<Tag color={"blue"|"default"}>{active|inactive}</Tag>` |
| last_delivery | `<Badge status>` + `<Tag color={STATUS_COLOR[s]}>` + timestamp снизу. NULL → серый «ещё не доставлялась». |
| created_at | timestamp в монотипном font'е |
| actions | `<Popconfirm><Button "Деактивировать" disabled={!active} /></Popconfirm>` |

**STATUS_COLOR** маппинг (совпадает с backend `DeliveryStatus` enum E9):
- OK → green (Badge.status="success")
- FAILED → red (Badge.status="error")
- RETRYING → orange (Badge.status="warning")

**CreateSubscriptionModal** — AntD `Modal + Form`:

| Поле | Тип | Валидация | Вкладывается в request |
|---|---|---|---|
| url | Input | required + AntD `type:"url"` (полная схема http://) | `url` (trim) |
| secret_id | Input | required | `secret_id` (trim) |
| filter.domains | Select mode="tags" tokenSeparators=[", "] | — | `filter.domains` (или null если пусто) |
| filter.codesets | Select mode="tags" | — | `filter.codesets` |
| filter.events | Select mode="multiple" options=[VERSION_PUBLISHED, VERSION_DEPRECATED] | — | `filter.events` |
| active | Switch (default ON) | — | `active` |

Filter: пустые arrays конвертируются в null перед отправкой — backend
интерпретирует null как «нет ограничения по этой оси» (handoff E9 §1.4).

**Mutations:**
- `createSubscription` → POST. На success: invalidate `qk.subscriptions.all()`
  + close modal + reset form. AntD `App.useApp().message` для UX-уведомлений.
- `deleteSubscription` → DELETE. На success: invalidate (row станет inactive).
  Запрещён UI на уже inactive строках (Popconfirm и Button disabled).

### 1.4. Frontend admin-guard внутри страницы

В дополнение к conditional Sider, сама `SubscriptionsPage` проверяет роль:

```tsx
if (!baseRoles.includes("RDM_ADMIN")) {
  return <Alert type="warning" message="Доступ запрещён" .../>;
}
```

Это нужно для случая, когда пользователь:
- зашёл по прямой ссылке `/admin/subscriptions`;
- relogin'улся под не-admin'ом, оставаясь на странице (Keycloak silent
  refresh может сменить роли).

Backend всё равно отдаст 403 на любой call к `/api/v1/subscriptions`, но
понятная UI-страница лучше, чем спам-`message.error("403: …")`.

### 1.5. Wire-types

```ts
export type SubscriptionEvent = "VERSION_PUBLISHED" | "VERSION_DEPRECATED";
export type DeliveryStatus = "OK" | "FAILED" | "RETRYING";

export interface SubscriptionFilter {
  domains?: string[] | null;
  codesets?: string[] | null;
  events?: SubscriptionEvent[] | null;
}

export interface Subscription {
  id: string;
  url: string;
  secret_id: string;
  filter: SubscriptionFilter | Record<string, unknown>;
  active: boolean;
  created_at?: string | null;
  created_by?: string | null;
  last_delivery_at?: string | null;
  last_delivery_status?: DeliveryStatus | null;
}
```

`filter: SubscriptionFilter | Record<string, unknown>` — defensive: backend
SubscriptionDto.View хранит filter как `Map<String, Object>` и при битом JSON
в БД отдаёт `{}` (handoff E9 §1.4 «fail-safe при битом JSON»). UI трактует
оба варианта одинаково через `<FilterBadge>` (он смотрит на конкретные ключи
domains/codesets/events).

## 2. Контракт: что UI ожидает от backend'а

### 2.1. Новое в round 2c (E9, RDM_ADMIN)

| Endpoint | Метод | Используется в | Auth |
|---|---|---|---|
| `/api/v1/subscriptions` | GET | SubscriptionsPage (list) | RDM_ADMIN |
| `/api/v1/subscriptions/{id}` | GET | (резерв на details-view) | RDM_ADMIN |
| `/api/v1/subscriptions` | POST | CreateSubscriptionModal | RDM_ADMIN |
| `/api/v1/subscriptions/{id}` | DELETE | SubscriptionsTable (deactivate) | RDM_ADMIN |

Семантика ошибок:
- `400` POST — invalid body (например, отсутствует `url`/`secret_id`). UI:
  `message.error` с описанием.
- `403` любой → frontend Alert «Доступ запрещён» (если страница уже
  открыта) или backend gate'ом без UI-fallback'а (Sider-пункта нет, по прямой
  ссылке покажется наша Alert).
- `404` GET/DELETE — subscription не существует. UI: `message.error`.
- `204` DELETE — успех (soft-delete; в response body ничего).

### 2.2. Без изменений

Все остальные endpoints — без изменений с round 2b. См.
[`E11.2b-ui-editor.md`](E11.2b-ui-editor.md) §2.

## 3. Что осталось доделать в E11.2c — мягкие follow-up'ы

Ничего не блокирует следующий шаг (V1+ feature audit viewer либо переход к E12
ingestion-коннектора). Список того, к чему стоит вернуться:

1. **Domain-scoped admin RBAC** (handoff E9 §6 #12, открытый вопрос команды
   банка). Сейчас один глобальный `RDM_ADMIN` управляет всеми subscription'ами.
   На пилоте ОК (1-2 admin'а, ≤10 subscription'ов). Когда дойдёт до десятков
   доменов — нужен либо `RDM_DOMAIN_ADMIN`-роль (asset-level в OM), либо
   фильтр на frontend'е по домену. Требует backend'а — это не UI-only.

2. **Manual replay для GIVE_UP outbox-rows** (handoff E9 §3 #2). Сейчас при
   `last_delivery_status="FAILED"` и `attempts>=8` admin видит только статус.
   Replay требует backend endpoint'а `POST /subscriptions/{id}/replay?event_id=...`
   — V1+. Когда появится — добавить кнопку «Replay» рядом с deactivate.

3. **GET /subscriptions/{id} не используется.** Отдельный detail-view не
   реализован, потому что вся информация умещается в строке таблицы. Если
   потребуется страница `/admin/subscriptions/{id}` (например, со списком
   последних outbox-events на эту subscription) — можно сделать через ещё
   один backend endpoint в V1+.

4. **Edit subscription** — backend не предоставляет PUT/PATCH endpoint'а.
   Backend pattern: чтобы изменить url/filter, надо deactivate старую +
   create новую. Это by-design (handoff E9 §1.9: «secret_id» — pointer; тело
   subscription'а должно быть immutable для аудитного следа). UI отражает это
   корректно.

5. **`Select mode="tags"` для domains/codesets** позволяет вводить любой
   qualified_name. Можно подгрузить список существующих domains/codesets из
   `api.listDomains()` + scan'а codesets, и предлагать как options. Минус —
   автокомплит может скрыть future-pattern'ы (subscription на ещё не созданный
   codeset). На пилоте — `tags`-режим достаточен.

6. **Filter validation на frontend'е.** Сейчас вводимые qualified_names —
   свободные строки. Backend SPEC требует lower-snake-case (handoff E9 §1.4).
   Можно добавить regex-валидацию в `Form.Item rules`. V1+ либо когда первый
   admin введёт `Risk Department` вместо `risk_department` и удивится, что
   фильтр не работает.

7. **Frontend не показывает количество застрявших outbox-rows.** Handoff E9
   §3 #7 предлагал backend healthcheck. Когда появится — можно вытащить в
   subscription row: «X messages stuck». Сейчас admin узнаёт только через
   `last_delivery_status="FAILED"`.

8. **`composite-key UX` через `key_spec.parts` и `ajv`-валидация attributes**
   (debt из E11.2a §3 #4 / E11.2b §3 #1-2). Не сделаны в этом round'е —
   focus был на subscriptions admin. Маленькие тех-улучшения, можно сделать
   отдельным мини-PR'ом независимо.

9. **`AntD bundle 331 KB gzip`** (был 325 в E11.2b — подросло на 6 KB от
   подтянутых Switch/Badge). Опт через `vite-plugin-imp` всё ещё актуален как
   V1+ задача.

10. **Не показывается время в локали пользователя.** `created_at` и
    `last_delivery_at` отображаются ISO-8601-строкой как есть. Можно
    подключить `dayjs` (он уже в AntD-зависимостях) и форматировать в
    локаль пользователя. Минор UX-улучшение.

## 4. Технический долг и решения, повлиявшие на следующие раунды

| Что | Где | Когда снять / следующий шаг |
|---|---|---|
| Frontend admin-guard через `useAuth().baseRoles` (не authoritative) | `SubscriptionsPage` + `AppLayout` | **Не снимать** — backend gate (`@RolesAllowed`) единственный authoritative. Frontend-check — UX-only. |
| `filter: SubscriptionFilter \| Record<string, unknown>` — defensive type | `types.ts:Subscription` | Если backend перестанет fail-safe-возвращать `{}` при битом JSON в БД — упростить до `SubscriptionFilter`. |
| Edit subscription не реализован (требует deactivate + create) | (отсутствует) | By-design backend; не реализовывать без PUT-endpoint'а. |
| Sider conditional через `MenuProps["items"]` сборку в useMemo | `AppLayout` | Стабильно — pattern легко расширяется на следующие admin-секции. |
| `"vendor-antd"` chunk +6 KB gzip от Switch/Badge | `vite.config.ts` | Acceptable; долгосрочная opt — `vite-plugin-imp`. |
| `qk.subscriptions.one(id)` не используется (нет detail-view) | `queryClient.ts` | Оставить — пригодится при V1+ extension'е. |

#
#   запустить полный 4-eyes flow на любом codeset'е → owner_approve → publish
#     → backend WebhookDeliveryWorker (5s tick) запостит в receiver
#     → через 10-15s обновить /admin/subscriptions:
#       last_delivery_status: OK (зелёный Badge + Tag)
#       last_delivery_at: ISO-8601 timestamp
#
#   тест RETRYING:
#     создать subscription с url=http://host.docker.internal:1/dead

Ожидаемые ошибки и UI:
- `400` POST (например, `url` без схемы) → AntD Form rule отлавливает до
  отправки; backend 400 → message.error.
- `403` любой → message.error («403: …»). Если пользователь оказался не на
  `/admin/subscriptions` — видит общий error toast.
- `404` DELETE (race: subscription уже deactivated в другой вкладке) →
  message.error + invalidate.

## 7. Открытые вопросы (актуальны для команды банка)

Без изменений с E11.2b, плюс:

1–24 — без изменений (см. `E11.2b-ui-editor.md` §7).
25. **Domain-scoped admin** (handoff E9 §6 #12) — пилотный объём (≤10
    subscription'ов на 2 домена) не требует, но для горизонта 3-5 лет
    (handoff §1.4 — десятки доменов) необходимо. Backend-задача.
26. **Edit subscription** — by-design нет PUT endpoint'а. Подтвердить с
    бизнесом, что deactivate+create — приемлемая модель change'ей.
27. **Auto-replay GIVE_UP outbox-rows** или manual через UI (V1+,
    backend-задача).
28. **Auth-claim group prefix.** UI-логика опирается на
    `baseRoles.includes("RDM_ADMIN")`. Если в проде Keycloak отдаёт префикс
    `/RDM_ADMIN` (зависит от group-mapper config'а), UI не распознает роль.
    Сейчас `extractBaseRoles` в `AuthContext.tsx` уже стрипает leading `/` —
    должно работать с обоими вариантами. Подтвердить с эксплуатацией prod-Keycloak'а.
