## 0. TL;DR

- **Было (F4).** `SubscriptionService.create` валидировал только схему
  `http(s)`; `WebhookDeliveryWorker` POST'ил подписанные payload'ы по
  любому host'у → SSRF в cloud-metadata / loopback / внутреннюю сеть /
  admin-порт. Митигаторы: admin-only + `followRedirects(NEVER)`.
- **Стало.** `EgressPolicy` (safe-by-default), проверка в двух точках:
  - **регистрация** (`SubscriptionService.create`) — best-effort
    fail-fast (резолвится-в-запрещённое → 400; нерезолвится → пропуск,
    авторитетен delivery-guard);
  - **доставка** (`WebhookDeliveryWorker.deliver`) — авторитетная
    **DNS-rebinding-safe** проверка (резолв прямо перед connect); блок
    = `GIVE_UP` без retry.
- **Политика (Q56).** loopback / any-local / multicast / link-local
  (вкл. `169.254.169.254`) — **hard-deny, не настраивается**. Приватные
  (10/8, 172.16/12, 192.168/16, 100.64/10, `fc00::/7`) — deny, если не
  в allowlist. Allowlist **пуст по умолчанию** → всё приватное запрещено.
  Публичные адреса разрешены (webhook наружу — штатный BR-15).
- **Конфиг банка (Q56).** `RDM_WEBHOOK_EGRESS_PRIVATE_ALLOWLIST` =
  CIDR'ы реальных consumer-сетей через запятую. Это единственное, что
  банк должен задать; код уже готов принять.

## 2. Дизайн-решения

- **Safe-by-default.** Пустой allowlist = всё приватное запрещено
  (строже = безопаснее). Метадата/loopback/link-local hard-deny без
  конфиг-override — легитимной нужды слать туда webhook нет, а это
  самый острый вектор.
- **DNS-rebinding-safe.** Авторитетная проверка — на доставке, резолв
  непосредственно перед connect (а не только при регистрации, где DNS
  мог быть другим). Registration-check — лишь быстрый UX-фидбэк.
- **Block = GIVE_UP, не RETRY.** Повтор на тот же запрещённый адрес
  бессмыслен; ретраи только зашумят outbox и отложат `FAILED`.
- **Конфиг через env** (как `EnvSigningKeyAdapter`/`ArchiveAdapters`) —
  `EgressPolicy.fromEnv()` в `PublishingModule.build`; сигнатуры наружу
  не меняются. Невалидный CIDR в конфиге → лог + игнор записи (политика
  остаётся строже, не «fail open»).
- **CIDR без зависимостей** (SPEC §3.1 lean) — байтовый префикс-матч
  v4/v6.

## 3. Что осталось — follow-up'ы

1. **Q56 — банк задаёт `RDM_WEBHOOK_EGRESS_PRIVATE_ALLOWLIST`.** Сейчас
   пусто → приватные consumer'ы (Risk-engine/IdM/BI в RFC1918) **не
   получат webhook'и**, пока CIDR не прописаны. Это сознательный
   safe-by-default; до прод-онбординга consumer'ов согласовать сетевую
   политику и заполнить env (compose/Helm).
2. **IT доставки под guard** (CI-only, Docker-Desktop §2 E14.9):
   подписка на приватный → outbox `GIVE_UP: egress …`, на публичный →
   доставка. По шаблону `extends PostgresIT`.
3. **Egress-метрика/алерт.** Счётчик `SSRF-BLOCKED` в delivery-логе →
   Prometheus + алерт (повтор блокировок = мисконфиг allowlist либо
   атака). V1+.
4. **DNS-pinning после connect.** Текущая реализация резолвит и
   проверяет перед `http.send`; теоретическое окно TOCTOU (resolve→
   фактический connect делает свой resolve). На JDK HttpClient без
   кастомного `Resolver`/socket-level pin закрыть полностью нельзя;
   риск низкий (followRedirects(NEVER) + короткий timeout). При
   ужесточении — кастомный `java.net.spi.InetAddressResolver` (JDK 18+)
   с pin на проверенный IP. Документировано.
