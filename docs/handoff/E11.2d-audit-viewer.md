## 1. Что сделано — backend

### 1.1. `AuditLogDao` (rdmmesh-audit/.../dao/AuditLogDao.java)

Добавлены два метода:

```java
@SqlQuery("""
    SELECT id, event_id, event_type, aggregate_type, aggregate_id, actor,
           occurred_at, payload::text AS payload
      FROM audit.audit_log
     WHERE (:eventType     IS NULL OR event_type     = :eventType)
       AND (:aggregateType IS NULL OR aggregate_type = :aggregateType)
       AND (:aggregateId   IS NULL OR aggregate_id   = :aggregateId)
       AND (:actor         IS NULL OR actor          = :actor)
       AND (:fromTs        IS NULL OR occurred_at   >= :fromTs)
       AND (:toTs          IS NULL OR occurred_at   <  :toTs)
       AND (:freeText      IS NULL OR payload->>'comment' ILIKE :freeText)
     ORDER BY occurred_at DESC, id DESC
     LIMIT :limit OFFSET :offset
""")
List<AuditEntry> findPaged(...);

// + аналогичный countFiltered с тем же WHERE
long countFiltered(...);
```

**Индексы** (миграция V070 не правится):
- `audit_log_event_type_ix(event_type, occurred_at DESC)` — покрывает
  `event_type=…` + ORDER BY.
- `audit_log_aggregate_ix(aggregate_type, aggregate_id, occurred_at DESC)` —
  покрывает aggregate-фильтры.
- `audit_log_actor_ix(actor, occurred_at DESC)` — покрывает actor-фильтр.
- `audit_log_payload_gin_ix(payload jsonb_path_ops)` — для `payload->>'comment'`
  free-text. ILIKE через GIN-индекс **не использует** его напрямую (нужен
  pg_trgm-индекс на текстовое выражение для substring); на пилоте объёмы
  небольшие, sequential scan по comment-полю быстрый. См. §3 follow-up #1
  про опт через trgm.

**Сортировка** — `(occurred_at DESC, id DESC)`. Tie-breaker по `id BIGSERIAL`
гарантирует stable pagination: два события с одинаковым `occurred_at`
(например, batch transitions в одной мс) не «прыгают» между страницами при
последовательных запросах.

### 1.2. `AuditResource` (rdmmesh-audit/.../resource/AuditResource.java) — новый

```java
@Path("/audit")
@Produces(MediaType.APPLICATION_JSON)
@RolesAllowed("RDM_ADMIN")
public final class AuditResource {

    @GET
    public Page list(
            @Auth RdmmeshPrincipal principal,
            @QueryParam("event_type") String eventType,
            @QueryParam("aggregate_type") String aggregateType,
            @QueryParam("aggregate_id") String aggregateIdRaw,
            @QueryParam("actor") String actorRaw,
            @QueryParam("from") String fromRaw,
            @QueryParam("to") String toRaw,
            @QueryParam("q") String freeText,
            @QueryParam("page") Integer page,
            @QueryParam("size") Integer size) { ... }

    public record Page(int page, int size, long total, List<AuditEntryDto> items) {}

    public record AuditEntryDto(
            long id, UUID eventId, String eventType,
            String aggregateType, UUID aggregateId, UUID actor,
            OffsetDateTime occurredAt, Map<String, Object> payload) {}
}
```

**Особенности:**
- `size` clamped в `[1, 1000]`, default 50. Большие страницы помогают
  consume'ить full audit для отчётности; UI ограничен `pageSizeOptions`
  до 500.
- `from`/`to` парсятся как `OffsetDateTime` (ISO-8601 с зоной). Невалидный
  формат → 400 с понятным message.
- `aggregate_id` / `actor` — UUID. Невалидный → 400.
- `q` (free-text) → ILIKE `%…%` по `payload->>'comment'`. Пользователь
  может видеть здесь reject-комменты, hotfix-причины, manual-deprecate-notes.
- Payload парсится на сервере через `ObjectMapper` (jsonb → `Map`). Не-объектный
  payload (массив или скаляр) заворачивается в `{value: …}` чтобы клиент
  всегда получал object-shape; невалидный JSON (теоретически невозможно
  благодаря jsonb-CHECK) → `{_raw: original}`.

### 1.3. `AuditModule.build` — новый Resources record

Раньше возвращал `AuditService` (он же сразу подписывался на EventBus
внутри). Теперь:

```java
public static Resources build(Jdbi jdbi, EventBus eventBus, ObjectMapper json) {
    AuditService service = new AuditService(jdbi, json);
    service.registerOn(eventBus);
    AuditResource resource = new AuditResource(jdbi, json);
    return new Resources(service, resource);
}

public record Resources(AuditService service, AuditResource resource) {}
```

`RdmmeshApplication.run` теперь:

```java
AuditModule.Resources audit = AuditModule.build(jdbi, eventBus, environment.getObjectMapper());
environment.jersey().register(audit.resource());
```

**Колонки:**
- `occurred_at` — `dayjs(...).format("YYYY-MM-DD HH:mm:ss")` (локальная зона).
- `event_type` — цветной Tag (blue/green/purple для известных типов; default
  для остальных).
- aggregate — `aggregate_type` Tag + copyable `aggregate_id` UUID.
- actor — copyable UUID.
- event_id — truncated `8 chars + …` + copyable.

**Pagination** — server-side через `page`/`size`. Backend отдаёт `total` →
AntD Table показывает корректный «Всего записей: N» внизу. При смене
фильтров `setPage(0)` (иначе можно оказаться на странице с нулевым total'ом).

**Expandable row** — payload pretty-printed JSON в `<pre>` с max-height 360px.
Свёрнут по умолчанию. `rowExpandable` отсекает события с пустым payload (не
случается на пилоте, но безопасно).

## 3. Что осталось доделать — мягкие follow-up'ы

Ничего не блокирует следующий шаг. Список того, к чему стоит вернуться:

1. **Free-text search через ILIKE — без trigram-индекса.** Сейчас
   `payload->>'comment' ILIKE %…%` идёт sequential scan по всем строкам,
   которые проходят остальные фильтры. На пилоте (≤тысячи rows) это OK; при
   росте — добавить миграцию V072:
   ```sql
   CREATE INDEX audit_log_comment_trgm_ix
     ON audit.audit_log USING gin ((payload->>'comment') gin_trgm_ops);
   ```
   pg_trgm уже подключён (V001 init).

2. **`RDM_AUDITOR`-роль** (handoff E10 §6 #15, открытый вопрос). Сейчас audit
   доступен только `RDM_ADMIN`. Compliance team, скорее всего, должна получить
   read-only роль без admin-прав на subscriptions. Требует:
   - добавить группу `RDM_AUDITOR` в `docker/keycloak/realms/realm-bank.json`
     + назначить тестовому юзеру;
   - изменить `@RolesAllowed` на `{"RDM_ADMIN", "RDM_AUDITOR"}`;
   - аналогично frontend-guard в `AuditPage` + Sider conditional.

3. **Audit-export в файлы.** Сейчас admin может только скроллить через UI.
   Полезен `GET /api/v1/audit/export?format=csv|json` — отдаёт `StreamingOutput`
   с тем же фильтром. Для compliance-аудитора (раз в год) — валидный
   use-case. V1+.

4. **Криптографическая hash-chain** (V14 / E14). Колонки `prev_hash` и
   `entry_hash` в `audit.audit_log` подготовлены ещё в V070, но не наполняются.
   План handoff E10 §3 #1: `entry_hash = sha256(prev_hash || event_id ||
   event_type || payload)`. Когда дойдёт — добавить миграцию V072 индекс на
   `(id ASC)`, расширить `AuditService.onEvent` либо batch-flusher, и UI
   verify-кнопку «Проверить целостность цепочки» (вызовет
   `GET /audit/verify-chain?from=…&to=…`).

5. **`payload` рендер только как JSON-string.** Можно добавить per-event_type
   custom rendering (например, для WORKFLOW_TRANSITION показать `from →
   to actor=…`). На пилоте JSON-pre достаточен, copy-paste-friendly.

6. **`X-Total-Count` header вместо `total` в body.** Сейчас `Page` record
   несёт total в JSON. Альтернатива — `X-Total-Count` header (RFC-style).
   На пилоте JSON-в-body проще для useQuery; не оптимизировать без причины.

7. **dayjs format в локали пользователя.** Сейчас формат hardcoded
   `YYYY-MM-DD HH:mm:ss`. Для admin'а это нормально (более компактно
   ISO-style), но можно подвязать `dayjs.locale(i18n.language)` для
   относительных форматов («2 минуты назад»). Минор.

8. **Vendor-antd chunk вырос на 37 KB gzip** от dayjs/dayjs-locales (DatePicker
   подтянул). Можно явно импортировать только нужные локали:
   `import "dayjs/locale/ru"; dayjs.locale("ru")` — отрезает английские/прочие
   локали. V1+ оптимизация.

9. **Testcontainers IT для audit endpoint'а** не написан. Backend сейчас
   покрыт unit'ом (`AuditEventClassifierTest`) + e2e smoke. Промежуточный
   слой (Resource → DAO → реальная БД с filtered query'ами) — `@DropwizardAppRule`
   + testcontainers Postgres. Тот же мягкий debt, что был у E3/E4/E5/E10.

10. **Streaming-export для очень больших window'ов.** Сейчас `findPaged`
    держит result в JVM heap; для размера 1000 это ~10 МБ JSON. Для V14
    (compliance hardening) добавится `GET /audit/stream?format=ndjson` через
    `ResultIterable.stream()` + `StreamingOutput`. На пилоте не нужно.

## 4. Технический долг и решения, повлиявшие на следующие раунды

| Что | Где | Когда снять / следующий шаг |
|---|---|---|
| `dropwizard-auth` теперь dep audit-модуля | `rdmmesh-audit/pom.xml` | **Не снимать.** Это правильное решение: REST-resource модуля живёт в самом модуле, не в app. |
| Free-text ILIKE без trgm-индекса | DAO + V070 | Когда первый admin пожалуется на медленный free-text — добавить trgm-индекс (см. §3 #1). |
| `RDM_ADMIN`-only audit access | `AuditResource @RolesAllowed` | Решить с командой compliance: достаточно ли admin'а или нужен отдельный AUDITOR (см. §3 #2). |
| `AuditEntryDto.payload: Map<String, Object>` | `AuditResource` | Парсинг на сервере — лишний CPU vs клиент. Альтернатива — отдавать raw JSON-string и парсить в JS. На пилоте серверный парсинг проще для клиента. |
| `Resources` record в `AuditModule` (раньше возвращался AuditService напрямую) | `AuditModule.build` | Стабильный pattern — повторяет PublishingModule.Resources (E9). |
| `audit` page key в Sider не объединён с subscriptions | `AppLayout` | Возможно вынести в const-array `ADMIN_SECTIONS` — micro-cleanup. |

```bash
#     прямой переход → Alert "Доступ запрещён" (frontend-guard,
#                                                useQuery disabled)
```

Ожидаемые ошибки:
- 400 на bad UUID в `aggregate_id`/`actor` → AntD-message с описанием.
- 400 на bad ISO timestamp в `from`/`to` → backend message; UI вряд ли
  допустит (DatePicker отдаёт valid ISO).
- 403 → frontend-guard перехватывает до запроса; backend 403 — fallback.
