## 0. TL;DR за 30 секунд

- **Проблема.** Ротация любого из трёх HMAC-секретов
  (`RDM_HMAC_KEY` / `RDM_OM_WEBHOOK_HMAC_KEY` / `RDM_WEBHOOK_KEY_*`) требовала
  одновременного переключения контрагента → окно отказа. Особо критичен
  inbound OM-webhook: его подписывает OM, проверяет RDM — рассинхрон секрета
  ронял **все** ownership/domain webhook'и в 401. Это блокировало prod-deploy
  (open question E6 §6 #4 / E7 §6 #4 / E9 §6 #4).

- **Решение — overlap-модель «primary + previous».**
  - **`SigningKeyPort.acceptedHmacKeys()`** (новый default-метод) — список
    ключей для *проверки*, primary первым. Подпись по-прежнему делается только
    `currentHmacKey()`.
  - **`WebhookKeyPort.resolveAllKeys(secretId)`** (новый default-метод) —
    симметрично для per-subscription ключей.
  - Default обоих = `[currentHmacKey()]` / `[resolveKey()]` —
    **backward-compatible**: лямбда-порты и моки, написанные до round 6,
    компилируются и работают как single-key.
  - **`EnvSigningKeyAdapter`** читает `<VAR>` (primary, с dev-fallback) и
    `<VAR>_PREVIOUS` (опциональный overlap, без fallback). Переработан с
    одного `byte[]` на `List<byte[]>`; добавлен ctor `(primary, previous)`.
  - **`EnvWebhookKeyAdapter.resolveAllKeys`** добавляет
    `RDM_WEBHOOK_KEY_<ID>_PREVIOUS` (фолбэк `RDM_WEBHOOK_KEY_DEFAULT_PREVIOUS`).
  - **`HmacVerifier.verify`** (E7 inbound) перебирает `acceptedHmacKeys()`
    без раннего выхода (`ok |= isEqual(...)`) — индекс совпавшего ключа не
    утекает по таймингу.

- **Wiring не менялся.** `RdmmeshApplication.fromEnv("RDM_HMAC_KEY", ...)` и
  `fromEnv("RDM_OM_WEBHOOK_HMAC_KEY", ...)` теперь *автоматически* подхватывают
  `*_PREVIOUS` — ротация чисто операционная (env), без передеплоя кода.

- **Процедура** — новый runbook `docs/runbooks/hmac-key-rotation.md`.

- **Backend-only round, без UI** (как и предсказывал E14.5 §7).

## 1. Что сделано

### 1.1. `rdmmesh-api` — rotation-примитив в портах

**`SigningKeyPort.java`**

```java
byte[] currentHmacKey();                       // sign + primary verify (как было)

default List<byte[]> acceptedHmacKeys() {      // НОВОЕ — verify против любого
    return List.of(currentHmacKey());          // default = single-key (BC)
}
```

**`WebhookKeyPort.java`**

```java
byte[] resolveKey(String secretId);            // sign primary (как было)

default List<byte[]> resolveAllKeys(String secretId) {  // НОВОЕ
    return List.of(resolveKey(secretId));      // default = single-key (BC)
}
```

Оба — `default`, поэтому SAM-лямбды (`() -> KEY.clone()` в тестах) и любые
сторонние реализации не требуют правок. Импорт `java.util.List` добавлен в оба
порта. ArchUnit чист: `java.util` уже разрешён в `rdmmesh-api`.

### 1.2. `EnvSigningKeyAdapter` — primary + `_PREVIOUS`

Переработан с `private final byte[] key` на `private final List<byte[]> keys`
(`[0]`=primary, `[1]`=previous если задан):

- `EnvSigningKeyAdapter(byte[])` — оставлен (delegate → `(key, null)`),
  backward-compatible.
- `EnvSigningKeyAdapter(byte[] primary, byte[] previous)` — новый; previous
  опционален, но если задан и < 32 байт → `IllegalArgumentException`
  (fail-fast на мисконфиге, не молча).
- `fromEnv(var, fallback)` теперь читает ещё `var + "_PREVIOUS"` (без
  fallback'а — overlap-ключ осмыслен только при реальной ротации).
- `currentHmacKey()` = `keys.get(0)`; `acceptedHmacKeys()` = defensive-copy
  всех. Клонирование на каждом выходе сохранено (как в исходнике).

### 1.3. `EnvWebhookKeyAdapter.resolveAllKeys`

`resolveKey` не тронут (primary, как было). `resolveAllKeys(secretId)`:
primary + `RDM_WEBHOOK_KEY_<UPPER>_PREVIOUS` (фолбэк
`RDM_WEBHOOK_KEY_DEFAULT_PREVIOUS`), та же `ensureLength` ≥32-проверка.

### 1.4. `HmacVerifier.verify` — accept-any-key (E7 inbound)

```java
boolean ok = false;
for (byte[] key : signingKey.acceptedHmacKeys()) {
    ok |= MessageDigest.isEqual(presented, compute(body, key));   // constant-time, без break
}
return ok;
```

Это **единственное изменение поведения** во всём round 6. Все прочие правки —
аддитивные (новые методы/ctor'ы/env-vars). `HmacSigner` (E6 sign) и
`OutboxOutboundAdapter` (E9 sign) не менялись — они и так подписывают primary.

### 1.5. `RdmmeshApplication` — без изменений wiring'а

Только комментарий у `omWebhookKey`-wiring'а: `fromEnv` уже сам подхватывает
`RDM_OM_WEBHOOK_HMAC_KEY_PREVIOUS`. Ротация — операционная (env + restart),
кода не трогает.

### 1.6. `HmacSigner` javadoc — зафиксировано E6-ограничение

Полная HMAC-перепроверка *исторических* `approval_signature` в verify-endpoint
**остаётся отложенной** (handoff E6 §3 #3): подписанный ISO-timestamp нигде не
персистится (`published_at` = SQL `now()` ≠ подписанный `Instant.now()`), и
воспроизвести подписанную строку нельзя без слома замороженной E6-формулы +
миграции. Round 6 даёт примитив (`acceptedHmacKeys()`), но потребитель для
E6-историч-verify — отдельный follow-up (см. §3 #1). Сегодня
`GET /versions/{id}/verify` и так сверяет только `content_hash` (SHA-256).

## 3. Что осталось — follow-up'ы (ничего не блокирует round 7)

1. **E6 historical-HMAC verify.** Чтобы verify-endpoint реально пересчитывал
   `approval_signature` (и работал под ротацией), нужно персистить подписанный
   timestamp. Варианты: (а) добавить колонку
   `code_set_version.signed_at timestamptz` и подписывать её же; (б) подписывать
   `published_at` из БД (читать после CAS в той же tx — инфраструктура
   round 5.1 это уже умеет, `findPublishedDetails(handle, ...)`). Вариант (б)
   меняет E6-формулу → миграция + переподпись/двойная-схема для старых версий.
   Отдельный round (предлагается **round 6.1** либо в составе round 8 security).
2. **Vault/SOPS-адаптеры.** `EnvSigningKeyAdapter`/`EnvWebhookKeyAdapter` —
   pilot-only. Prod: `VaultSigningKeyAdapter` с теми же
   `currentHmacKey()`/`acceptedHmacKeys()` (Vault versioned secret: current =
   latest, previous = N-1). Drop-in, бизнес-логику не трогает.
3. **Авто-снятие overlap.** Сейчас снятие `_PREVIOUS` — ручной шаг runbook'а.
   Можно добавить TTL-метаданные на ключ + healthcheck «overlap активен > N
   суток» (оператор забыл свернуть ротацию = расширенная attack-surface).
4. **Smoke под реальным OM.** §5 не прогонялся в этой сессии (нужен `make up`).

## 4. Технический долг и решения

| Что | Где | Когда снять / следующий шаг |
|---|---|---|
| `acceptedHmacKeys()`/`resolveAllKeys()` — default single-key | ports | by-design BC; rotation-aware только в env-адаптерах. |
| Перебор всех ключей в `HmacVerifier` (нет early-exit) | `HmacVerifier.verify` | by-design — constant-time-ish; 1–2 ключа, число не секретно. |
| `_PREVIOUS` без dev-fallback | `EnvSigningKeyAdapter.fromEnv` | by-design — overlap осмыслен только в реальной ротации; в dev не нужен. |
| E6 historical-HMAC verify не закрыт | `HmacSigner`/`PublishingService.verify` | §3 #1 — нужен persisted signed-timestamp + миграция. |
| Env-адаптеры вместо Vault | `*KeyAdapter` | §3 #2 — prod заменяет адаптер. |
| `fromEnv` читает env напрямую (не тестируется без env) | unit-тесты | конструкторный путь покрыт; env-путь — как у `EnvWebhookKeyAdapterTest` (fallback-only). |

### 5.2. Zero-downtime ротация inbound OM-webhook

```bash
OLD=rdmmesh-dev-om-webhook-key-change-me-in-prod-vault   # текущий dev-fallback
NEW=$(openssl rand -base64 48)
WEBHOOK=http://localhost:8080/api/v1/webhooks/om/ownership
sign(){ python3 -c "import sys,hmac,hashlib;print('sha256='+hmac.new(sys.argv[1].encode(),sys.stdin.buffer.read(),hashlib.sha256).hexdigest())" "$1"; }

# (1) baseline: подпись старым ключом проходит (overlap НЕ развёрнут)
B='{"event_id":"rot-1","entity_type":"domain","entity_id":"99999999-9999-9999-9999-999999999999","fully_qualified_name":"treasury","owners":[],"experts":[],"reviewers":[]}'
curl -s -o /dev/null -w '%{http_code}\n' -XPOST "$WEBHOOK" -H "X-OM-Signature: $(printf %s "$B"|sign "$OLD")" --data-binary "$B"   # 200

# (2) развернуть overlap: RDM_OM_WEBHOOK_HMAC_KEY=$NEW, RDM_OM_WEBHOOK_HMAC_KEY_PREVIOUS=$OLD
#     (env в docker/.env или compose override) → make up заново.

# (3) во время overlap проходят ОБА ключа:
B2='{"event_id":"rot-2",...}' ; B3='{"event_id":"rot-3",...}'
curl ... -H "X-OM-Signature: $(printf %s "$B2"|sign "$NEW")" --data-binary "$B2"   # 200 (new)
curl ... -H "X-OM-Signature: $(printf %s "$B3"|sign "$OLD")" --data-binary "$B3"   # 200 (previous)

# (4) снять overlap: убрать RDM_OM_WEBHOOK_HMAC_KEY_PREVIOUS → restart.
B4='{"event_id":"rot-4",...}'
curl ... -H "X-OM-Signature: $(printf %s "$B4"|sign "$OLD")" --data-binary "$B4"   # 401 (старый больше не принят)
curl ... -H "X-OM-Signature: $(printf %s "$B4"|sign "$NEW")" --data-binary "$B4"   # 200
```

Полная пошаговая процедура (E6/E7/E9, откат, чек-лист) —
