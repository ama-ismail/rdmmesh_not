## 0. TL;DR

- **Топология per Domain.** RDM_ADMIN деплоит BPMN на домен →
  `tenantId=domainId` (Flowable native multi-tenancy).
  `FlowableWorkflowEngine` при старте инстанса резолвит домен версии
  (version→codeset→domain) и берёт активный tenant-процесс домена;
  нет шаблона / сбой резолва → дефолтный `rdm4eyes`.
- **No-bypass сохранён (модель A).** Делегат каждого шага по-прежнему
  зовёт `WorkflowService.transition` → enum-`StateMachine` (self-approval
  /role-gate). Кастомный BPMN не может перепрыгнуть 4-eyes (доказано в IT:
  кастомный процесс домена + self-approval всё равно 409).
- **Контракт шаблона** (`BpmnTemplateValidator`, deploy-time):
  обязателен receive-task `rt_await` + service-task
  `delegateExpression=${rdmTransitionDelegate}`; иначе 400, в Flowable
  ничего не попадает.
- **Реестр/аудит** `workflow.workflow_template` (V032,
  append-only-по-смыслу): домен → process_key, flowable_deployment_id,
  bpmn_sha256, version, deployed_by/at, active (1 активный/домен).
  Воспроизводимость (SPEC §2.3).
- **REST** (RDM_ADMIN, только при `engine=flowable`):
  `POST /api/v1/workflow/domains/{id}/template` (BPMN bytes),
  `GET …/template` (активный), `GET /workflow/templates` (все),
  `DELETE …/template` (откат к дефолту).

## 2. Контракт / поведение

- `engine=enum` → REST шаблонов не зарегистрирован; всё как раньше.
- `engine=flowable`, домен без шаблона → дефолтный `rdm4eyes`
  (round 1) без изменений.
- Деплой шаблона: BPMN с process-id `X`, обязательными `rt_await` +
  `${rdmTransitionDelegate}`. Активна последняя версия; `DELETE`
  снимает active (Flowable tenant-деплой остаётся, но не используется).
- Кастомная топология может добавлять/переименовывать задачи, listener'ы,
  шлюзы вокруг якорей — но каждый фактический переход валидируется
  enum-StateMachine (модель A). Прыгнуть в обход 4-eyes нельзя.
