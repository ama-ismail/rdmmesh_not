## 0. TL;DR за 30 секунд

- **Проблема (E14-compliance §3 #7, E13.3 §3 #6).** Два admin/observability-
  действия не попадали в append-only audit:
  - `GET /api/v1/audit/verify-chain` — кто/что/исход проверки целостности.
  - `POST /api/v1/versions/{id}/closure/rebuild` — DR-перестройка closure
    (раньше только `log.warn` в сервисе).
- **Решение.** Два новых `DomainEvent`-типа + классификация + эмиссия на
  тот же in-process bus, который слушает `rdmmesh-audit` (E10). Зацикливания
  нет — отдельные `event_type`, эмиссия из GET/POST по запросу пользователя,
  не из записи в журнал.
- **Новые event_type'ы:** `AUDIT_VERIFY_CHAIN` (aggregate `AUDIT`,
  actor=кто звал), `CLOSURE_REBUILD` (aggregate `VERSION`/versionId,
  actor=admin).
- **Backend-only**, без UI, без миграций, без изменения схемы. Контракт
  REST не изменён (verify-chain/rebuild возвращают то же).

## 2. Контракт / поведение

- **REST не изменён.** `verify-chain` и `closure/rebuild` возвращают те же
  тела. Добавлен только side-effect: append одной audit-row'и.
- **Цикл исключён.** `AUDIT_VERIFY_CHAIN`-row пишется `AuditService`'ом с
  `id > toId` уже-вычисленного диапазона → на ответ verify-chain не влияет;
  следующий полный verify-chain включит её как обычную валидную chain-row
  (она записана штатным hash-chain путём). verify-chain — GET по запросу,
  записью в журнал не триггерится.
- **Best-effort publish.** Обе эмиссии в try/catch с `log.warn` — сбой
  in-process шины не должен ломать verify-chain (GET) или DR-rebuild
  (SPEC §3.8: audit append-failure не блокирует бизнес-операцию; тут даже
  слабее — это observability).
- **payload.** Спец-POJO у новых событий нет (внутренние observability,
  не REST/webhook-контракт). `AuditService.extractPayload` для незнакомых
  типов сериализует сам record → payload =
  `{eventId,occurredAt,actor,...}`. `AuditService` усекает `occurred_at`
  до µs (E14.7 фикс B) единообразно — precision-drift не воспроизводится.

## 3. Что осталось — follow-up'ы

1. **IT на эмиссию (CI-only).** Дедицированный IT «verify-chain →
   AUDIT_VERIFY_CHAIN-row в audit_log», «rebuild → CLOSURE_REBUILD-row» —
   по шаблону `HashChainRoundtripIT extends PostgresIT`. Локально скипнётся
   (Docker-Desktop, E14.9 §2), реально пройдёт на CI. Не написан в этом
   round'е: классификация покрыта unit'ом (9 тестов), эмиссия —
   compile-checked + best-effort паттерн как у `WorkflowService`; полный
   roundtrip — когда CI-gating подтверждён (E14.9 §3 #4).
2. **R9-backlog `AtomicRollbackIT`** (E14.9 §3 #2) — по-прежнему открыт,
   осознанно: требует прогона на CI/dockerd.
3. **CI-gating** (E14.9 §3 #4) — проверить при первом push, что push-job
   `verify` без `-DskipITs`.
4. **UI (опц.).** AuditPage может показывать новые `event_type`'ы в
   фильтре — это уже есть автоматически (фильтр свободный по event_type),
   отдельной доработки не требует. Цветовой бейдж для `AUDIT_VERIFY_CHAIN` —
   косметика V1+.
5. **verify-chain на пустом журнале.** Early-return (minId==null) НЕ
   эмитит событие — verify пустого журнала тривиален. Если compliance
   захочет фиксировать и «звонок по пустому журналу» — добавить эмиссию
   в early-return ветку. Сейчас by-design (минимум шума).

## 4. Технический долг и решения

| Что | Где | Решение |
|---|---|---|
| Эмиссия из resource-слоя, не из сервиса | AuditResource / ClosureAdminResource | by-design — минимально инвазивно; сервисы остаются чистыми (AuthoringService без EventBus) |
| `AuthoringModule.build` +param EventBus | сигнатура | единственный caller — RdmmeshApplication, обновлён; тест-callers нет |
| Нет IT на эмиссию | — | §3 #1 (CI-only, как весь R9-backlog) |
| verify-chain на пустом журнале без события | AuditResource | §3 #5 by-design |
| Каждый verify-chain пишет audit-row | AuditService | by-design — это и есть требуемое event-coverage (E14-compliance §3 #7); объём мал (admin/auditor-операция, не hot-path) |
