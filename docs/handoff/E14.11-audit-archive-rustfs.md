## 0. TL;DR за 30 секунд

- **Проблема.** SPEC §3.8: audit-сегменты должны уходить в immutable-store
  до дропа партиции (retention). E14.7 §3 #2:
  `drop_audit_partition_if_archived(p_partition, p_archived, …)` доверял
  `p_archived` как честному аргументу — факт архива не проверялся.
- **Решение (round 10).**
  - `ArchivePort` (rdmmesh-api) — storage-agnostic; адаптер
    `RustFsArchiveAdapter` (minio-client) в composition-root'е
    (rdmmesh-audit не имеет права на S3-клиент — ArchUnit).
  - V074: `audit.archive_manifest` (immutable: INSERT/SELECT,
    REVOKE U/D/T) + `audit.assert_segment_archived(text)` +
    **1-арг** `audit.drop_audit_partition_if_archived(text)` — выводит
    archived ИЗ манифеста, делегирует в замороженную V073 3-арг функцию
    (retention-window/DEFAULT/not-found не дублируются).
  - `AuditArchiveService` (rdmmesh-audit): месячный сегмент → ndjson
    (детерминированно, переиспользует `AuditExportWriter`) → SHA-256 →
    `ArchivePort.putImmutable` → INSERT в `archive_manifest`.
  - `POST /api/v1/audit/archive?year=&month=` (`RDM_ADMIN`).
  - RustFS добавлен в dev-compose (S3 :9000 / console :9001).
- **Disabled-by-default.** Пустой `RDM_ARCHIVE_ENDPOINT` → no-op stub
  ({@code ArchivePort.enabled()=false}), сервис без RustFS не падает
  (паттерн OM-lookup E2). Архивация — опциональный ops-flow, не hot-path.
- **Backend + infra.** UI не тронут. Миграция V074 auto-apply.

## 2. Дизайн-решения и риски

- **RustFS вместо S3/MinIO** (выбор пользователя): Apache-2.0, без AGPL/
  платных ограничений, drop-in S3. minio-java работает против любого
  S3-API → vendor-agnostic за `ArchivePort` (как SPEC §3.3 ports&adapters).
- **Object-Lock — ПОДТВЕРЖДЁН рабочим на RustFS** (smoke §5:
  `retentionApplied:true`). Bucket создаётся `objectLock(true)`, на
  объект — GOVERNANCE retention до `month_end + 10y` (SPEC §3.7); RustFS
  (`rustfs/rustfs:latest`, 2026-05) принял S3 `PutObjectRetention`.
  Адаптер всё равно деградирует мягко (если иная сборка/режим store'а
  отвергнет retention → объект записан, `retention_applied=false` +
  WARNING) — **честно, без молчаливой потери WORM**; но в проверенной
  конфигурации WORM реально работает.
  `assert_segment_archived` при `retention_applied=false` даёт RAISE
  WARNING (не ошибку) — дроп не блокируется, но след в логе есть.
- **Drop-guard link.** 1-арг `drop_audit_partition_if_archived(text)` —
  единственный «безопасный» путь: archived из манифеста. Замороженная
  V073 3-арг функция не менялась (retention-window/DEFAULT/not-found
  логика переиспользована делегированием).
- **Детерминизм.** Сегмент = помесячная партиция, имя совпадает с
  `ensure_audit_partition` V073 (`audit_log_yYYYYmMM`) → `segment_label`
  матчится drop-guard'ом. ndjson rows ORDER BY id ASC, snapshotMaxId
  фиксирован → SHA-256 воспроизводим offline-аудитором.
- **Манифест immutable.** INSERT/SELECT + REVOKE U/D/T на rdmmesh_app
  (та же дисциплина, что audit_log V070). Повторная архивация →
  `ON CONFLICT (segment_label) DO NOTHING` (идемпотентно).
- **Тег образа `rustfs/rustfs:latest`** — для dev ок; для prod
  запинить версию (follow-up §3 #1).

### 5.2. Независимая верификация RustFS (не самоотчёт сервиса)

Проверено сторонним S3-клиентом `minio/mc` (контейнер на
`rdmmesh_default`), не через rdmmesh:

```
on-disk:  /data/rdmmesh-audit-archive/audit/2026/05/audit_log_y2026m05.ndjson/xl.meta
mc ls:    5.3KiB STANDARD audit/2026/05/audit_log_y2026m05.ndjson
mc stat:  X-Amz-Object-Lock-Mode=GOVERNANCE, Retain-Until=2036-06-01T00:00:00Z
mc retention info: GOVERNANCE, expiring in 3667 days
mc cat | sha256sum = f939668573d9be78ddb900bd04d42023e458be491bf0d2465064d6e405ca6cd9
                     == archive_manifest.content_sha256   (бит-в-бит)
```

Вывод: объект физически в RustFS (диск + S3 API); Object-Lock GOVERNANCE
до month_end+10y реально выставлен и виден сторонним клиентом; содержимое
идентично захэшированному сервисом. WORM-immutability подтверждена
end-to-end независимо.

> **Дефект, пойманный smoke'ом (исправлен).** `findExportPage` с непустыми
> `fromTs/toTs`: Postgres `could not determine data type of parameter $6` —
> типизированный nullable-параметр, чей первый инстанс `:p IS NULL`,
> оставлял parser без типа. Латентно с E14.4 (export не передавал даты).
> Фикс: явные `CAST(:p AS timestamptz|uuid)` для fromTs/toTs/aggregateId/
> actor в `findExportPage` (заодно харднит E14.4-export). После фикса —
> smoke зелёный.

## 6. Открытые вопросы (для команды банка)

Без изменений с E14.10 (Q56–60), плюс:
61. **RustFS prod-топология.** Single-node vs distributed; бэкап самого
    RustFS; где хранить root-creds (Vault). Архитектор + DBA/DevOps.
62. **Object-Lock режим.** GOVERNANCE (admin override) vs COMPLIANCE
    (необратимо до retain_until). Регулятор IFRS9 может требовать
    COMPLIANCE. Уточнить с compliance перед prod.
63. **Retention горизонт архива.** Сейчас month_end+10y (строже из
    SPEC §3.7 7/10). Per-event-type разделение 7/10 — Q54 (E14.7).
