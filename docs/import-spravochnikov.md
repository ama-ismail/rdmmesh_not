# Импорт справочника в rdmmesh из Excel / CSV

> Памятка для бизнес-пользователей (Author): как подготовить файл для bulk-импорта
> записей (CodeItem) в DRAFT-версию справочника. Контракт колонок **общий** для
> Excel (.xlsx) и CSV — различается только формат файла.

## 1. Куда и кто грузит

- Импорт доступен на странице **версии справочника в статусе DRAFT**, кнопка
  **«Bulk-импорт»** → вкладка **Excel (XLSX)**, **CSV** или **JSON**.
- Нужна роль **Author** (или Admin) в домене справочника.
- Загрузка **атомарна**: если хоть одна строка не прошла валидацию — отклоняется
  **весь** файл (статус `REJECTED`), в справочник не пишется ничего. Исправьте
  ошибки из таблицы и загрузите заново.
- Импорт делает **upsert** по `key_parts`: новая запись добавляется, существующая
  (с тем же ключом) — обновляется.

## 2. Структура файла Excel

- **Первая строка — заголовок** с именами колонок. Со второй строки — данные.
- Имена колонок матчатся **без учёта регистра**; лишние пробелы, NBSP, BOM
  вокруг имени игнорируются. То есть `key_parts`, `Key_Parts`, `KEY_PARTS` —
  одно и то же.
- Лист: парсер берёт **первый лист, где в заголовке есть `key_parts`**. Можно
  держать первым лист «Инструкция» — данные найдутся на следующем.
- Полностью пустые строки пропускаются (можно оставлять «для красоты»).
- Колонок может быть меньше: обязательна только `key_parts`, остальные
  опциональны и могут отсутствовать.

CSV — то же самое: первая строка заголовок, разделитель запятая, значения с
запятыми/кавычками — в двойных кавычках по RFC 4180.

## 3. Колонки

| Колонка | Обяз. | Что писать |
|---|---|---|
| `key_parts` | **да** | Ключ записи. Одиночный — просто значение: `KZ`. Композитный (матрицы) — JSON-массив строкой: `["RETAIL","BB","12M"]`. |
| `label_ru` | нет | Наименование (рус). |
| `label_en` | нет | Наименование (англ). |
| `description_ru` | нет | Описание (рус). |
| `description_en` | нет | Описание (англ). |
| `parent_key` | нет | Родитель для иерархии внутри справочника. Формат как у `key_parts` (одиночный или JSON-массив). |
| `attributes` | нет | Все атрибуты одним JSON-объектом строкой: `{"pd":0.0234,"stage":"2"}`. Числа/булевы сохраняют тип. Валидируется по схеме справочника (CodeSetSchema). |
| `attr.<имя>` | нет | Один атрибут в отдельной колонке (UI-friendly). Напр. колонка `attr.stage` со значением `2`. Имя атрибута регистрозависимо. |
| `order_index` | нет | Целое — порядок сортировки. |
| `status` | нет | `ACTIVE` (по умолчанию) или `RETIRED`. |
| `effective_from` | нет | Дата начала действия, ISO `ГГГГ-ММ-ДД`. |
| `effective_to` | нет | Дата окончания действия, ISO `ГГГГ-ММ-ДД`. |

`parent_ref` (cross-codeset ссылки) через Excel/CSV в текущей версии не
задаётся — только через API/JSON.

## 4. Атрибуты: `attributes` против `attr.<имя>` (важно)

Атрибуты записи валидируются по **CodeSetSchema** справочника (типы, enum,
обязательность). Есть два способа их передать:

1. **`attributes`** — JSON-объект строкой. Типы берутся из JSON как есть:
   `{"pd":0.0234,"active":true,"stage":"2"}` → число, булево, строка.
   **Для числовых атрибутов используйте этот способ.**
2. **`attr.<имя>`** — по колонке на атрибут. Коэрция значения **намеренно
   ограничена**: только `true`/`false` → булево; **всё остальное — строка**.
   Числа здесь НЕ распознаются специально — иначе ломается enum-валидация
   вида `stage ∈ {"1","2","3"}` (где `1` — строка, а не число).

Практика: строковые/enum-атрибуты и флаги — удобно колонками `attr.*`; числовые
(ставки, PD/LGD, коэффициенты) — в одной колонке `attributes` JSON-строкой.
Способы можно смешивать; при совпадении имени значение из `attr.*`
перекрывает то, что в `attributes`.

## 5. Примеры

### 5.1. Одиночный ключ, атрибуты колонками

| key_parts | label_ru | label_en | attr.stage |
|---|---|---|---|
| S1 | Стадия 1 | Stage 1 | 1 |
| S2 | Стадия 2 | Stage 2 | 2 |

### 5.2. Композитный ключ + числовой атрибут JSON-строкой

| key_parts | label_ru | attributes |
|---|---|---|
| `["RETAIL","BB","12M"]` | Розница BB 12м | `{"pd":0.0234}` |
| `["RETAIL","BB","24M"]` | Розница BB 24м | `{"pd":0.0312}` |

### 5.3. Иерархия + даты действия

| key_parts | parent_key | label_ru | effective_from | status |
|---|---|---|---|---|
| DEP-1 | | Департамент 1 | 2026-01-01 | ACTIVE |
| DIV-1 | DEP-1 | Отдел 1 | 2026-01-01 | ACTIVE |

## 6. Частые ошибки

| Сообщение / симптом | Причина и что делать |
|---|---|
| `column 'key_parts' is required` | В строке пустой ключ — заполните `key_parts`. |
| `Ни на одном листе нет … 'key_parts'` (со списком листов и заголовков) | Не та колонка/лист. Сверьте имя колонки с разделом 3; данные должны быть на листе, где заголовок содержит `key_parts`. |
| `attributes do not match CodeSetSchema` | Значение/тип атрибута не подходит под схему справочника (enum, тип, обязательность). Для чисел — через `attributes` JSON, не `attr.*`. |
| `cannot parse key_parts '...'` | Композитный ключ должен быть валидным JSON-массивом строк: `["A","B"]`. |
| `... must be ISO date (YYYY-MM-DD)` | Дата не в формате `2026-01-31`. В Excel задайте текстовый формат ячейки или пишите дату строкой. |
| Импорт `REJECTED`, ничего не загрузилось | Это by design: пачка атомарна. Исправьте все ошибки из таблицы и повторите. |
